## 目标与非目标

### 目标

修复用户反馈的 8 个体验缺陷：

| #   | 问题                                            | 根因（一句话）                                                     |
| --- | ----------------------------------------------- | ------------------------------------------------------------------ |
| 1   | 配对码过期时 PairDialog 被 AgentFormDialog 遮挡 | `.dialog-backdrop` z-index=100 < `.dialog-drawer` 1000             |
| 2   | 大任务超时 + "上游服务不可用" 错误不可排查      | gateway → core 硬编码 15s；core → LLM 硬编码 60s；502 文案丢上下文 |
| 3   | 发送后只 disable 按钮，无任何运行反馈           | ChatDialog 仅 `disabled`，缺 spinner/计时/取消                     |
| 4   | 单次回复 tokens 上限 32000 不合理               | client zod + input + core zod + db CHECK 四处硬编码                |
| 5   | API Key 非必填，配置易遗漏                      | client zod `default('')`、UI 标 "optional"                         |
| 6   | 重启客户端丢失配对（**真根因**）                | `secure-store.ts` 写 Stronghold 后未调 `stronghold.save()`         |
| 7   | 中文模式下主题按钮显示 LIGHT/DARK               | `zh-CN/translation.json` 里 `theme.light/dark` 是英文              |
| 8   | 主页 / 配对弹窗多处 ASCII 英文文案未中文化      | zh-CN 里残留 "GATEWAY PAIR / REMOTE PAIRING & AUTH / REQ / OPT" 等 |

### 非目标

- **不改 LLM 流式（SSE）协议**：取消按钮仅取消客户端 fetch，gateway 仍跑完。
- **不改 contextWindow 字段（保持 2_000_000 上限）**：用户反馈仅针对 maxCompletionTokens。
- **不引入 silent re-pair 机制**：Stronghold 落盘修好后用户预期已满足。
- **不删除 cleanup 任务、不动 TTL 默认值**：与 #6 根因无关，TTL 是次要决定。
- **不动后端 API Key 字段（保持 nullable + env fallback）**：#5 明确仅前端必填，兼容老数据 + Ollama 等无 key 场景。

## 2. 超时可配 + 默认值放宽 + 错误文案带原因（Issue #2）

### 2.1 timeout 拓扑

```
client fetch (无 timeout)
  └→ gateway → core      : CORE_TIMEOUT_MS = 640_000 (10min40s)
        └→ core → LLM    : LLM_TIMEOUT_MS  = 600_000 (10min)
```

实际只有两个 env：`CORE_TIMEOUT_MS`（gateway 持有 core 连接的上限）与 `LLM_TIMEOUT_MS`（core 调 LLM 的 AbortSignal 上限）。core 自身没有第三层 timeout — 它收到 gateway 请求后只调 LLM，等待时间就是 LLM_TIMEOUT_MS。

CORE_TIMEOUT_MS 比 LLM_TIMEOUT_MS 大 40s 缓冲，覆盖 gateway↔core RTT + core 解析响应包装等开销，避免 gateway 先于 core 切断进行中的 LLM 调用。

### 2.2 gateway

```diff

+  CORE_TIMEOUT_MS: z.coerce.number().int().positive().default(640_000),
```

```diff
// gateway/src/server.ts
-  const core = new CoreClient({ baseUrl: cfg.CORE_URL });
+  const core = new CoreClient({ baseUrl: cfg.CORE_URL, timeoutMs: cfg.CORE_TIMEOUT_MS });
```

```diff
// gateway/.env.example
+ # gateway → core 上游 HTTP 客户端超时（ms）。
+ # 必须 > core/.env 里的 LLM_TIMEOUT_MS（建议 20s 缓冲），否则 gateway 会先于 core 切断 LLM 调用。
+ # 默认 640000 = 10min40s，覆盖大多数 LLM 思考密集型任务。
+ CORE_TIMEOUT_MS=640000
```

## 3. ChatDialog 发送中：spinner + 计时 + 取消（Issue #3）

### 3.1 状态扩展

```diff
 type ChatState =
   | { kind: 'idle' }
-  | { kind: 'sending'; pending: { tempId: string; content: string } }
+  | { kind: 'sending'; pending: { tempId: string; content: string }; startedAt: number }
   | { kind: 'error'; pending: { tempId: string; content: string } | null; error: string };
```

`startedAt = Date.now()`；onSubmit 时 set；error/success 时 reset。

### 3.2 计时刷新

```typescript
// state.kind === 'sending' 时启动 1s tick，让 UI 重渲示 elapsedSec
const [tick, setTick] = useState(0);
useEffect(() => {
  if (state.kind !== 'sending') return;
  const id = setInterval(() => setTick(t => t + 1), 1000);
  return () => clearInterval(id);
}, [state.kind]);
const elapsedSec = state.kind === 'sending' ? Math.floor((Date.now() - state.startedAt) / 1000) : 0;
```

### 3.3 取消

```typescript
const abortRef = useRef<AbortController | null>(null);

const onCancel = () => {
  abortRef.current?.abort();
  dispatch({ type: 'send-error', error: t('chat.cancelled') });
};

// onSubmit 内：
abortRef.current = new AbortController();
const res = await postMessage(gatewayUrl, clientKey, sid, content, abortRef.current.signal);
```

`postMessage` 签名扩 `signal?: AbortSignal`，透传给 `apiFetch`；`apiFetch` 把 `signal` 透传给 `fetch`。

### 3.4 按钮 UI

```jsx
<button type="submit" disabled={!input.trim() || state.kind === 'sending'} ...>
  {state.kind === 'sending' ? (
    <>
      <span className="spinner" aria-hidden="true" />
      {t('chat.sending')} {elapsedSec}s
    </>
  ) : (
    t('chat.send')
  )}
</button>
{state.kind === 'sending' && (
  <button type="button" onClick={onCancel} className="btn">
    {t('chat.cancel')}
  </button>
)}
```

```typescript
// 新结构：const SCHEMA_VERSION = 5;
// 检查 schema_version 时：
//   - 不存在 → 新库走 schema.sql + INSERT version=5
//   - 已存在 version < SCHEMA_VERSION → 顺序 apply migrations[version..SCHEMA_VERSION-1]
//   - 已存在 version > SCHEMA_VERSION → loud fail（降级风险）
//   - 已存在 version === SCHEMA_VERSION → ok

const MIGRATIONS: Record<number, (db: DatabaseType) => void> = {
  // 4 → 5: 解除 max_tokens 上限（SQLite CHECK 不能 ALTER，需要表重建）
  4: db => {
    db.exec(loadMigrationSql('0005_relax_max_tokens.sql'));
  },
};

function loadMigrationSql(name: string): string {
  const url = new URL(`./migrations/${name}`, import.meta.url);
  return readFileSync(fileURLToPath(url), 'utf8');
}
```

**core/src/db/migrations/0005_relax_max_tokens.sql**（新建文件；agents 表完整字段已对照 schema_version=4 的 schema.sql 复制）：

```sql
-- 4→5: 解除 agents.max_tokens 上限（≤32000 → 仅保留 ≥1）。
-- SQLite 不支持 ALTER CONSTRAINT，需表重建。所有字段与 schema_version=4 一致，
-- 仅 max_tokens CHECK 被放宽。
BEGIN TRANSACTION;

CREATE TABLE agents_new (
  id                TEXT    PRIMARY KEY,
  name              TEXT    NOT NULL UNIQUE,
  description       TEXT    NOT NULL DEFAULT '',
  llm_provider      TEXT    NOT NULL DEFAULT 'openai-compatible',
  base_url          TEXT    NOT NULL,
  model             TEXT    NOT NULL,
  max_tokens        INTEGER,
  context_window    INTEGER,
  reasoning_effort  TEXT,
  api_key           TEXT,
  enabled_api       INTEGER NOT NULL DEFAULT 0,
  system_prompt     TEXT    NOT NULL DEFAULT '',
  capabilities      TEXT    NOT NULL DEFAULT '[]',
  created_at        TEXT    NOT NULL,
  updated_at        TEXT    NOT NULL,
  CHECK (llm_provider = 'openai-compatible'),
  CHECK (max_tokens IS NULL OR max_tokens >= 1),                              -- ← 放宽，去掉 ≤32000
  CHECK (context_window IS NULL OR (context_window >= 1 AND context_window <= 2000000)),
  CHECK (reasoning_effort IS NULL OR reasoning_effort IN ('none', 'minimal', 'low', 'medium', 'high', 'xhigh')),
  CHECK (length(name) > 0 AND length(name) <= 64),
  CHECK (length(description) <= 256),
  CHECK (length(model) > 0 AND length(model) <= 128),
  CHECK (length(base_url) > 0 AND length(base_url) <= 512),
  CHECK (length(system_prompt) <= 8192),
  CHECK (enabled_api IN (0, 1))
);

INSERT INTO agents_new SELECT * FROM agents;
DROP TABLE agents;
ALTER TABLE agents_new RENAME TO agents;

-- 重建 agents 表上的 index（schema_version=4 仅一个 idx_agents_name）
CREATE UNIQUE INDEX IF NOT EXISTS idx_agents_name ON agents(name);

UPDATE schema_version SET version = 5, applied_at = strftime('%Y-%m-%dT%H:%M:%fZ','now');

COMMIT;
```

## 6. Stronghold 落盘（Issue #6）—— 真根因修复

### 6.1 Bug 复述

`secure-store.ts` 当前 `saveToTauri` 写入 `store.insert()` 后未调 `stronghold.save()`。Stronghold 设计：`store.insert` 只改内存，必须 `stronghold.save()` 才写磁盘。所以重启后磁盘文件里没有 CONFIG_KEY，`loadFromTauri` 返回 null，App 进入 NEED_PAIR。

### 6.2 修复

```typescript
// client/src/lib/secure-store.ts

async function getOrCreateClient(stronghold: Stronghold) {
  // Stronghold v2 plugin: createClient 在 record 已存在时抛错。
  // 先 loadClient，失败再 createClient（首次启动时）。
  try {
    return await stronghold.loadClient(RECORD_NAME);
  } catch {
    return await stronghold.createClient(RECORD_NAME);
  }
}

async function saveToTauri(cfg: SecureConfig): Promise<void> {
  const stronghold = await Stronghold.load(STORE_PATH, STORE_PASSWORD);
  const client = await getOrCreateClient(stronghold);
  const store = client.getStore();
  const bytes = new TextEncoder().encode(JSON.stringify(cfg));
  await store.insert(CONFIG_KEY, Array.from(bytes));
  await stronghold.save(); // ← 关键：把 in-memory 改动写回磁盘
}

async function loadFromTauri(): Promise<SecureConfig | null> {
  const stronghold = await Stronghold.load(STORE_PATH, STORE_PASSWORD);
  const client = await getOrCreateClient(stronghold);
  const store = client.getStore();
  const bytes = await store.get(CONFIG_KEY);
  if (!bytes) return null;
  const json = new TextDecoder().decode(bytes);
  return JSON.parse(json) as SecureConfig;
}

async function clearFromTauri(): Promise<void> {
  const stronghold = await Stronghold.load(STORE_PATH, STORE_PASSWORD);
  const client = await getOrCreateClient(stronghold);
  const store = client.getStore();
  await store.remove(CONFIG_KEY);
  await stronghold.save(); // ← 同上
}

// 删除原 getClient 辅助函数
```

## 11. 风险与回滚

| 风险                                             | 缓解                                                                                                                     |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| #6 Stronghold loadClient/createClient 兼容性问题 | 单测覆盖；如果 Tauri 新版 plugin API 变 → 降级到完全 `createClient` + ignore "exists" 错误                               |
| #4 migration 重建表数据丢失                      | 在事务内 + INSERT 全行；如发现 schema.sql 字段拷漏 → migration 失败 loud fail，用户回滚到旧二进制即可（DB version 仍 4） |
| #2 LLM_TIMEOUT_MS 过大 → 用户感受 "卡死"         | UI 已有 #3 计时显示 + 取消；用户随时可主动放弃                                                                           |

回滚策略：每个 phase 独立 commit，可单独 revert。#4 DB migration 一旦升到 v5，**降级二进制将 loud-fail**（schema 版本高于代码期望）；如必须降级需手动 `UPDATE schema_version SET version=4`（约束放宽 → 旧 CHECK 仍兼容老数据）。
