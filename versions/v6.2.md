## 二、目标 / 非目标

### 目标

- **gateway 端职责清晰**：纯透传 + v3 鉴权；不存 agent / session / message 数据；10 个端点（v3 现状 GET `/v1/agents` 改造 + v6.2 新增 9 个）按 core 端响应整包回包。
- **完整透传 9 错码**：core 端 v6.1 细化的 9 个业务码（`ok` / `invalid_body` / `unauthorized` / `agent_not_found` / `session_not_found` / `agent_name_conflict` / `internal_error` / `not_implemented` / `upstream_error`）由 gateway 整包透传，HTTP status 也透传；client 看到与 core 完全一致的 `{data, code, message}`。
- **`X-Internal-Client-Key` 内部 header**：gateway→core 透传时挂上，值 = `req.clientCtx.id`（v3 middleware 已挂，事实为 sha256 hash）；不重哈希、不解析。
- **v3 鉴权层零改动**：v3 SHA-256 middleware 不动；`req.clientCtx.id` 字段（v3 现状已有）继续作为内部 header 源。
- **路由文件拆 3**：`gateway/src/routes/agents.ts`（v3 现状文件名沿用，内容扩 4 端点）/ `sessions.ts`（新增，3 端点）/ `messages.ts`（新增，2 端点）。
- **CoreClient 扩展 9 方法**：在 v3 现状 `health` / `listAgents` 基础上加 9 个方法（`createAgent` / `getAgent` / `updateAgent` / `deleteAgent` / `createSession` / `getSession` / `deleteSession` / `listMessages` / `postMessage`）；统一走 `private call()` 辅助挂内部头。
- **双层测试**：路由层单测（mock `CoreClient`）+ integration 测（mock core HTTP server）；覆盖 10 端点 × 错误码矩阵 + 头注入 + 网络异常。
- **v3 + v6.1 兼容**：v3 client / v3 gateway / v3 core 完全无感（v6.2 增的路径对 v3 透明）；v6.1 client / v6.1 core 完全无感（v6.2 gateway 透传 9 端点 = v6.1 期望的入口）。
- **零网络协议破坏**：v6.2 改的是"加新端点" + "内部 header 注入"；`X-Client-Key` 公网鉴权、响应 `{data, code, message}` 包装、compat 握手全部不变。

### 非目标

- **不**实现其他鉴权策略：v6.2 完全沿用 v3 SHA-256 middleware；不引入 JWT / OAuth / mTLS 等。
- **不**做 LLM 调用：gateway 仅透传到 core；任何 LLM 业务逻辑都在 core（v6.1 已实现）。
- **不**做端点级缓存：v6.2 每次请求都打 core；GET `/v1/agents` 不缓存（YAGNI；后续 评估）。
- **不**做全透传框架：v6.2 手写 10 个 handler（每个端点 5-10 行）；未来若端点数翻倍再考虑 `core-proxy` 通用代理。
- **不**做限流 / metrics：v6.2 无；后续 评估 per-clientKey 限流 + Prometheus `/metrics`。
- **不**做 client 端"虚拟办公室" UI：v6.2 是网关层设计；client 改造由后续 v6.3+ 承接。
- **不**改 v6.1 已实现的 core 端任何行为：v6.1 9 错码 / 10 端点 / DB schema / LLM factory 全部不动；v6.2 仅是"补上 v6.md 中 gateway 部分的实施级细化"。
- **不**做 stream / SSE / WebSocket 推送：v6.2 仍为同步透传；streaming 后续 评估。
- **不**改 compat 矩阵范围：v6.2 仅涉及 gateway 单端代码改动，**预留 v6.2 实际写代码时由实施 plan 按 v2 规则 bump gateway 版本**（v6.2 文档不锁 bump 数字）。
- **不**改 v3 / v6.1 任何已有功能的行为：v3 鉴权 / pair / 内部路由 / v6.1 core 端所有端点全部保留。
- **不**重写 v6.1 决策 16："明文存"措辞保持 v6.1 原状；v6.2 文档在 5.8 兼容节明确"v6.2 实施后 sessions.client_key 实际存 hash"的差异，留待 v6.1 文档后续修订时统一。
- **不**复述 v6.1 已锁定的 9 错码含义：v6.2 仅说"gateway 透传 9 错码"；错码触发条件 / 错误码细化 / 字段约束以 v6.1 §5.3.3 / §5.7.2 为准。

**v3 现状 `agents.ts` 路由**（`gateway/src/routes/agents.ts`）：

```ts
// v3 现状：仅 GET /v1/agents
export async function agentRoutes(app: FastifyInstance, core: CoreClient) {
  app.get('/v1/agents', async (req, reply) => {
    try {
      const result = await core.listAgents();
      return reply.send(ok(result));
    } catch (e) {
      req.log.error({ err: e }, 'listAgents failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });
}
```

**v3 现状 `CoreClient`**（`gateway/src/clients/core.ts`）：

- 2 个方法：`health()` / `listAgents()`
- 用 `undici.request()`（不是 Node 内置 fetch）
- `listAgents()` **不**接收 `clientKey` 入参（v3 阶段 gateway→core 透传时不带任何 clientKey 头）

**v3 现状 `authMiddleware`**（`gateway/src/auth/middleware.ts`）：

- 验证 `X-Client-Key`（SHA-256 比对 DB 中 `key_hash` 字段）
- 验证通过后挂 `req.clientCtx = { id: client.id, name: client.name }`
- `client.id` 实际是 sha256 hash（看 `gateway/src/auth/store.ts:92-96`：`commitPairingCode` 把 `keyHash` 同时塞进 `id` 和 `key_hash` 两列；`findByHash` 注释虽叫 findByHash 但 SQL 是 `WHERE id = ?`）
- **原始明文 key 在 middleware 验证后被丢弃**

**v6.1 core 现状**（v6.1 已实现；v6.2 不动）：

- 10 个新端点（5 agents + 3 sessions + 2 messages）走 `app.decorate('db', ...)` + DAO
- `internalClientKeyHook` 读 `X-Internal-Client-Key` → `req.internalClientKey`；缺失 → 401
- 路由取 `req.internalClientKey` 写 `sessions.client_key`（`core/src/routes/sessions.ts:40-47`）
- core 端 hook 对 header 值**无感**（`core/src/hooks/internal-client-key.ts:25-29`：直接 `req.internalClientKey = ck`）
- 9 错码统一走 `app.setErrorHandler` 转 `{data: null, code, message}`（v6.1 §5.7.1）

**v6.md 现状**（commit `5ab702d`）：

- v6 是**三端全景**设计（client / gateway / core 一起）
- gateway 部分描述分布在 5.1-5.5 + 第六章变更清单 + 第八章待处理
- v6 gateway 部分**未锁**的粒度（v6.2 要补的）：
  - `X-Internal-Client-Key` header **值语义**（v6.md 写"挂 clientKey"未明说是明文还是 hash）
  - **错误码透传粒度**（v6.md 5.3 错码表是 v1 风格的 5 个；v6.1 core 端细化为 9 个）
  - **路由文件拆分**（v6.md 第六章 "gateway/src/routes/agents.ts" 写改造 + 多个 ts 透传，未锁具体拆几份）
  - **CoreClient 方法签名**（v6.md 第六章 "加 9 个 core client 方法" 未列方法签名 / 入参顺序）
  - **错误码透传策略**（v6.md 5.3 错码表里 `upstream_error` 502 涵盖 v6.1 的 502 场景；v6.2 要确认 v6.1 的 9 错码如何被 gateway 区分透传）

**v3 → v6.2 代码层面变化**（gateway 端）：

```
v3: buildServer() 注册 agentRoutes(instance, core)
    core = new CoreClient({ baseUrl }) —— 2 方法，不挂内部 header
v6.2: buildServer() 注册 agentRoutes / sessionRoutes / messagesRoutes 三块
       core = new CoreClient({ baseUrl }) —— 11 方法（health + 10 业务）
       所有业务方法统一走 core.call(method, path, clientKey, body) 挂 X-Internal-Client-Key
```

**v3 → v6.2 数据流变化**（client→gateway→core，发消息为例）：

```
v3  POST /v1/chat:
   client → gateway (X-Client-Key) → 透传 → core → 同步返回
   v3 阶段无 /v1/sessions/{id}/messages 路由；v6.1 core 已实现，gateway 未透传

v6.2  POST /v1/sessions/{sid}/messages:
   client → gateway (X-Client-Key 鉴权 → req.clientCtx.id = sha256)
        → core (X-Internal-Client-Key: <sha256 hash>) 内部透传
        → core 写 sessions.client_key = sha256 hash（v3 现状事实，与 v6.1 决策 16 措辞小差异）
        → core LLM.chat() → 同步返回 { userMessage, assistantMessage }
        → gateway 整包回包 {data, code:0, message:'ok'}（不解析 data）
        → client 收到 200 + {data, code:0, message:'ok'}
```

**鉴权矩阵**：

| 链路           | 鉴权机制                                         | 失败处理                                     |
| -------------- | ------------------------------------------------ | -------------------------------------------- |
| client→gateway | `X-Client-Key` + v3 middleware                   | 401 `missing_key` / `invalid_key`（v3 沿用） |
| gateway→core   | `X-Internal-Client-Key: <sha256 hash>`           | 401 `unauthorized`（v6.1 core hook 拒绝）    |
| 公开路由       | 白名单前缀（`/health` / `/pair` / `/internal/`） | middleware 不处理                            |
| 内部路由       | handler 自检 `req.ip === '127.0.0.1'`            | 拒访                                         |

**约束**：

- v6.2 **不**改 v3 鉴权 middleware；`req.clientCtx.id` 字段（v3 现状已有）继续作为 gateway→core 头源
- 内部 `X-Internal-Client-Key` 不暴露给公网；仅 gateway 注入
- v6.2 透传层不参与权限判断（v6.1 决策 16 跨 clientKey 共享）

### 5.3 核心数据流：发一条消息

```
client (v6.3+ 虚拟办公室，agent 弹窗)
  │
  │ POST /v1/sessions/{sid}/messages
  │ header: X-Client-Key: <clientKey>      (v3 鉴权，原始明文)
  │ body:   { id: <umId>, content: "..." }
  │
  ▼
gateway (v3 鉴权 + v6.2 透传):
  1. authMiddleware 验 X-Client-Key（SHA-256 比对）
     ├─ 缺失 → 401 missing_key
     ├─ hash 不在 DB → 401 invalid_key
     └─ 通过 → req.clientCtx = { id: sha256(clientKey), name }
  2. handler 入口取 ck = req.clientCtx.id (= sha256 hash)
  3. core.postMessage(ck, sid, req.body) 内部调 core.call('POST', path, ck, body)
     └─ call() 内置：
        - headers: { 'Content-Type': 'application/json', 'X-Internal-Client-Key': ck }
        - method: POST
        - body: JSON.stringify(body)
  4. 拿 core 响应：{ status, data } （data = core 返回的 .data 字段）
  5. handler 包装：reply.code(status).send(ok(data))
     ├─ 成功 → 200 + {data, code:0, message:'ok'}
     └─ 失败（core 4xx/5xx）→ 原 status + ok(data)（data 为 null 也整包透传）
  6. 异常处理：core.call 抛网络错 → catch → 502 + err(502, 'upstream_error')
  │
  ▼
core (v6.1 已实现):
  1. internalClientKeyHook 读 X-Internal-Client-Key → req.internalClientKey
  2. 验 body (zod) → { id, content }
  3. DB.sessions.get(sid) → 拿 agent_id
  4. DB.agents.get(agent_id) → 拿 9 字段
  5. DB.messages.listBySession(sid) → 拼上下文
  6. OpenAICompatibleLLMClient.chat(...) → 拿 reply
  7. DB insert user message (id = umId) + assistant message (id = amId)
  8. 同步返回 { data: { userMessage, assistantMessage }, code: 0, message: 'ok' }
  │
  ▼
client 收到：HTTP 200 + { data: {userMessage, assistantMessage}, code: 0, message: 'ok' }
```

**关键约定**：

- gateway **不**解析 `data` 字段内容（不知道 userMessage / assistantMessage 是什么）
- gateway **不**重写 `code` / `message`（透传 core 的业务码）
- gateway **不**改 HTTP status（透传 core 的 200/4xx/5xx）
- 鉴权失败由 gateway 自己抛（v3 middleware），不打到 core
- 网络层异常（core 不可达 / 超时）由 gateway `call()` 抛出 → handler catch → 502 `upstream_error`

```ts
// 上游 core 的 HTTP 客户端：基于 undici，比 Node 内置 fetch 更轻、控时更细。
// v6.2：扩 9 个方法 + 统一 call() 辅助挂 X-Internal-Client-Key 头。
// clientKey 入参 = sha256 hash（v3 middleware req.clientCtx.id 字段；不做重哈希）。
import { request } from 'undici';

export type CoreClientOptions = {
  baseUrl: string;
  timeoutMs?: number;
};

export class CoreClient {
  private readonly baseUrl: string;
  private readonly timeoutMs: number;

  constructor(opts: CoreClientOptions) {
    this.baseUrl = opts.baseUrl.replace(/\/+$/, '');
    this.timeoutMs = opts.timeoutMs ?? 15_000;
  }

  /**
   * 统一调用入口：挂 X-Internal-Client-Key 头 + 处理 HTTP 状态 / body 透传。
   * - 2xx：返回 { status, data: <.data 字段或 null> }
   * - 4xx/5xx：原 status 透传 + data = <.data 字段>（gateway handler 整包回包）
   * - 网络错：throw Error（handler catch → 502 upstream_error）
   */
  private async call(
    method: 'GET' | 'POST' | 'PATCH' | 'DELETE',
    path: string,
    clientKey: string,
    body?: unknown,
  ): Promise<{ status: number; data: unknown }> {
    const res = await request(`${this.baseUrl}${path}`, {
      method,
      headers: {
        'Content-Type': 'application/json',
        'X-Internal-Client-Key': clientKey,
      },
      headersTimeout: this.timeoutMs,
      bodyTimeout: this.timeoutMs,
      body: body !== undefined ? JSON.stringify(body) : undefined,
    });

    const status = res.statusCode;
    // 204 No Content：body 为空
    if (status === 204) return { status, data: null };
    // 其他：解析外层 { data, code, message }，取 .data
    const json = (await res.body.json()) as { data?: unknown };
    return { status, data: json.data ?? null };
  }

  /**
   * 上游健康检查：5xx 视为 down，避免 core 启动中导致网关"假绿"。
   * v6.2 保留 v3 现状签名（不挂内部头；/health 是白名单）。
   */
  async health(): Promise<{ ok: boolean; service: string }> {
    const res = await request(`${this.baseUrl}/health`, {
      method: 'GET',
      headersTimeout: this.timeoutMs,
      bodyTimeout: this.timeoutMs,
    });
    if (res.statusCode >= 500) {
      return { ok: false, service: 'core' };
    }
    return (await res.body.json()) as { ok: boolean; service: string };
  }

  /**
   * 列上游 agents：v3 现状签名扩展，接收 clientKey 走 call() 挂内部头。
   * 返回 { status, data: Agent[] }（data 可能为 null 当 4xx/5xx）。
   */
  async listAgents(clientKey: string): Promise<{ status: number; data: unknown[] }> {
    return this.call('GET', '/v1/agents', clientKey) as Promise<{
      status: number;
      data: unknown[];
    }>;
  }

  // === agents CRUD（v6.2 新增）===
  async createAgent(clientKey: string, body: unknown): Promise<{ status: number; data: unknown }> {
    return this.call('POST', '/v1/agents', clientKey, body);
  }
  async getAgent(clientKey: string, id: string): Promise<{ status: number; data: unknown }> {
    return this.call('GET', `/v1/agents/${encodeURIComponent(id)}`, clientKey);
  }
  async updateAgent(
    clientKey: string,
    id: string,
    body: unknown,
  ): Promise<{ status: number; data: unknown }> {
    return this.call('PATCH', `/v1/agents/${encodeURIComponent(id)}`, clientKey, body);
  }
  async deleteAgent(clientKey: string, id: string): Promise<{ status: number; data: unknown }> {
    return this.call('DELETE', `/v1/agents/${encodeURIComponent(id)}`, clientKey);
  }

  // === sessions CRUD（v6.2 新增）===
  async createSession(
    clientKey: string,
    body: unknown,
  ): Promise<{ status: number; data: unknown }> {
    return this.call('POST', '/v1/sessions', clientKey, body);
  }
  async getSession(clientKey: string, id: string): Promise<{ status: number; data: unknown }> {
    return this.call('GET', `/v1/sessions/${encodeURIComponent(id)}`, clientKey);
  }
  async deleteSession(clientKey: string, id: string): Promise<{ status: number; data: unknown }> {
    return this.call('DELETE', `/v1/sessions/${encodeURIComponent(id)}`, clientKey);
  }

  // === messages（v6.2 新增）===
  async listMessages(
    clientKey: string,
    sessionId: string,
  ): Promise<{ status: number; data: unknown }> {
    return this.call('GET', `/v1/sessions/${encodeURIComponent(sessionId)}/messages`, clientKey);
  }
  async postMessage(
    clientKey: string,
    sessionId: string,
    body: unknown,
  ): Promise<{ status: number; data: unknown }> {
    return this.call(
      'POST',
      `/v1/sessions/${encodeURIComponent(sessionId)}/messages`,
      clientKey,
      body,
    );
  }
}
```

```ts
// gateway/src/routes/agents.ts（v6.2 改造 + 4 端点）
import type { FastifyInstance, FastifyRequest } from 'fastify';
import { ok, err } from '../response.js';
import type { CoreClient } from '../clients/core.js';

export async function agentRoutes(app: FastifyInstance, core: CoreClient) {
  // 公共：取 clientKey 的小辅助（v3 middleware 已挂；类型上可能为 undefined，做运行时断言）
  const getClientKey = (req: FastifyRequest): string => {
    if (!req.clientCtx) throw new Error('clientCtx not set (auth middleware not run?)');
    return req.clientCtx.id;
  };

  // 1. GET /v1/agents（v3 现状路径，v6.2 改造走 call()）
  app.get('/v1/agents', async (req, reply) => {
    const ck = getClientKey(req);
    try {
      const { status, data } = await core.listAgents(ck);
      return reply.code(status).send(ok(data));
    } catch (e) {
      req.log.error({ err: e }, 'listAgents failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });

  // 2. POST /v1/agents
  app.post('/v1/agents', async (req, reply) => {
    const ck = getClientKey(req);
    try {
      const { status, data } = await core.createAgent(ck, req.body);
      return reply.code(status).send(ok(data));
    } catch (e) {
      req.log.error({ err: e }, 'createAgent failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });

  // 3. GET /v1/agents/{id}
  app.get<{ Params: { id: string } }>('/v1/agents/:id', async (req, reply) => {
    const ck = getClientKey(req);
    try {
      const { status, data } = await core.getAgent(ck, req.params.id);
      return reply.code(status).send(ok(data));
    } catch (e) {
      req.log.error({ err: e, id: req.params.id }, 'getAgent failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });

  // 4. PATCH /v1/agents/{id}
  app.patch<{ Params: { id: string } }>('/v1/agents/:id', async (req, reply) => {
    const ck = getClientKey(req);
    try {
      const { status, data } = await core.updateAgent(ck, req.params.id, req.body);
      return reply.code(status).send(ok(data));
    } catch (e) {
      req.log.error({ err: e, id: req.params.id }, 'updateAgent failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });

  // 5. DELETE /v1/agents/{id}
  app.delete<{ Params: { id: string } }>('/v1/agents/:id', async (req, reply) => {
    const ck = getClientKey(req);
    try {
      const { status, data } = await core.deleteAgent(ck, req.params.id);
      return reply.code(status).send(ok(data));
    } catch (e) {
      req.log.error({ err: e, id: req.params.id }, 'deleteAgent failed');
      return reply.code(502).send(err(502, 'upstream_error'));
    }
  });
}
```

### 5.5 错误码透传矩阵

#### 5.5.1 透传策略

- **core 2xx 响应**：`reply.code(status).send(ok(data))` —— 成功码 0 + data 透传
- **core 4xx/5xx 响应**：`reply.code(status).send(ok(data))` —— HTTP status 透传 + data 整包透传（data 为 null 时也是合法的 `{data: null, code, message}`）
- **gateway 网络层异常**（core 不可达 / 超时）：`reply.code(502).send(err(502, 'upstream_error'))`
- **gateway 路由内部异常**：被 `app.setErrorHandler` 兜底为 `reply.code(500).send(err(500, 'internal_error'))`
- **v3 鉴权失败**：middleware 直接 `reply.code(401).send(err(401, 'missing_key' / 'invalid_key'))`，不打到 core

#### 5.5.2 9 错码 × 10 端点矩阵

| 触发                          | core 返回                                                    | gateway 行为                                                 | client 看到                                                          |
| ----------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | -------------------------------------------------------------------- |
| 成功                          | 200 + `{data, code:0, message:'ok'}`                         | `reply.code(200).send(ok(data))`                             | 200 + `{data, code:0, message:'ok'}`                                 |
| zod 校验失败                  | 400 + `{data:null, code:400, message:'invalid_body'}`        | 整包透传                                                     | 400 + `{data:null, code:400, message:'invalid_body'}`                |
| 内部 header 缺失              | 401 + `{data:null, code:401, message:'unauthorized'}`        | 整包透传（理论上不该出现：gateway 必挂）                     | 401 + `{data:null, code:401, message:'unauthorized'}`                |
| agent 不存在                  | 404 + `{data:null, code:404, message:'agent_not_found'}`     | 整包透传                                                     | 404 + `{data:null, code:404, message:'agent_not_found'}`             |
| session 不存在                | 404 + `{data:null, code:404, message:'session_not_found'}`   | 整包透传                                                     | 404 + `{data:null, code:404, message:'session_not_found'}`           |
| name 重复                     | 409 + `{data:null, code:409, message:'agent_name_conflict'}` | 整包透传                                                     | 409 + `{data:null, code:409, message:'agent_name_conflict'}`         |
| LLM upstream 不可达           | 502 + `{data:null, code:502, message:'upstream_error'}`      | 整包透传                                                     | 502 + `{data:null, code:502, message:'upstream_error'}`              |
| core 进程挂 / 网络层异常      | （core 不会主动返回 5xx；网络错）                            | gateway `call()` 抛 → handler catch → 502 + `upstream_error` | 502 + `{data:null, code:502, message:'upstream_error'}`              |
| gateway 路由内部异常          | gateway 自己抛                                               | setErrorHandler → 500 + `internal_error`                     | 500 + `{data:null, code:500, message:'internal_error'}`              |
| v3 鉴权失败（client→gateway） | （gateway 自身；不打到 core）                                | middleware 401 missing_key / invalid_key                     | 401 + `{data:null, code:401, message:'missing_key' / 'invalid_key'}` |

#### 5.5.3 `not_implemented` 501（v6.1 预留）

- v6.1 core 端保留 `not_implemented` 501 码值，但 `agents.llm_provider` CHECK 约束仅 `'openai-compatible'`，实际不触发
- v6.2 gateway 透传策略与 502 同：整包透传（理论场景：未来 v6.3+ 加新 provider 但未实现）
- 当前 v6.2 集成测试**不**覆盖（`not_implemented` 在 v6.2 阶段不可触发；保留码值）

#### 5.5.4 透传原则汇总

- core 的 HTTP status / 业务码 / message 字符串**全部整包透传**，不重写
- gateway 这一侧**仅**捕获**网络层异常**（core 不可达 / 超时），包装为 502 `upstream_error`
- v3 鉴权失败（401 missing_key / invalid_key）由 gateway 自身产生，**不**走 core
- gateway 路由 handler 不应解析 `data` 字段；解析是 client 的事
