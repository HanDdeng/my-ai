## 一、问题陈述

v3 合入后，三个子项目（client / gateway / core）共积累 32 个测试文件，全部与源码"贴身存放"在 `src/` 树内：

```
client/src/App.test.tsx                    ← 对应 src/App.tsx
client/src/compat/handshake.test.ts        ← 对应 src/compat/handshake.ts
gateway/src/auth/store.test.ts             ← 对应 src/auth/store.ts
gateway/src/routes/internal/pair-resolve.test.ts  ← 2 层深
core/src/agent/registry.test.ts            ← 对应 src/agent/registry.ts
...
```

这种"贴近源码"的布局是 v1 起步时为减少路径复杂度做的最小选择，跑了一年下来暴露三个问题：

- **测试代码与生产代码物理混在一起**。`src/` 里既有可发布的业务代码，也有 200~300 行的测试用例；阅读源码时易被 `.test.ts` 干扰；做发布构建（`tsc -b`）时也会被一并编译进 typecheck 流（虽然 `noEmit` / `rootDir: src` 拦住了产物输出，但语义上不干净）。
- **`client/src/test/` 已经是半成品例外**。`setup.ts` 单独放在 `src/test/`，但所有其他测试文件都贴身放在 `src/auth/`、`src/components/` 等真实模块下，结构不一致——既不是"全贴近"也不是"全分离"。
- **导入路径用相对深度，未来挪文件易坏**。32 个测试里 20 个含相对 import，深度从 `../` 到 `../../` 不等；以后想把某个测试连同被测模块一起迁到 `src/foo/`，每条 import 都要重新算深度。

v4 的任务：**把 32 个测试文件统一迁出 `src/`，放到每个子项目根的 `test/` 树下，与 `src/` 形成完全镜像的目录结构；导入方式从"相对深度"切换到 tsconfig path alias `@/*`，让测试位置与被测代码位置解耦。**

## 二、目标 / 非目标

### 目标

- **结构镜像**：`<subproject>/test/<path>` 与 `<subproject>/src/<path>` 严格一一对应；任何 `src/foo/bar.ts` 的测试都叫 `test/foo/bar.test.ts`。
- **src 净化**：`src/` 树内**不再有任何 `*.test.ts` / `*.test.tsx`**，也不再有 `test/` 子目录。
- **导入去深度化**：测试中的被测代码 import 全部从 `'../foo/bar.js'` 等相对路径切到 `'@/foo/bar'`，深度恒为 0。
- **vitest 与 tsconfig 同步指向新位置**：三套 `vitest.config.ts` 的 `include` / `setupFiles` 改为 `test/...`；三套 `tsconfig.json` 的 `include` 加上 `test`。
- **git 友好**：用 `git mv` 而非 add+delete，保留 rename history。
- **零行为变更**：v4 不改任何被测代码、不改任何测试断言、不改任何 coverage 阈值；只动文件位置和导入写法。

### 非目标

- **不调整被测代码组织**：`src/` 内部的目录结构（`auth/` / `compat/` / `routes/` / `components/` / `lib/`）v4 不动。
- **不引入新的测试结构**（不拆 E2E、integration、unit 三层；不分 spec/unit/contract；不引入 fixtures 目录）。
- **不引入新的测试工具**（不装 `vite-tsconfig-paths`、不加 coverage threshold、不加 mutation testing）。
- **不补缺失的测试**（v3 §八移交的待办与 v4 正交，留给后续版本）。
- **不调整 client 的 `src-tauri/`**：Tauri 端的 Rust 代码、tauri.conf.json、capabilities 不动。
- **不 bump 任何版本号**：v4 是纯架构维护；`package.json` / `versions/compat-matrix.json` 不动。

## 四、方案对比

brainstorming 阶段依次否决/选定的关键路径：

| 阶段              | 备选                                                                      | 选定                                                                  |
| ----------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| 目标位置          | `<subproject>/test/...` / 顶层 `tests/<sub>/...` / 保留 `src/test/...`    | **`<subproject>/test/...`**（与 src 镜像；用户决定）                  |
| setup 文件位置    | 跟总体 `test/setup.ts` / `test/setup/setup.ts` / 保留 `src/test/setup.ts` | **`test/setup.ts`**（跟总体结构一致）                                 |
| 导入方式          | 相对深度改写 / tsconfig path alias / `rootDirs`                           | **tsconfig path alias `@/*`**（tsconfig 已配，零成本）                |
| 迁移方式          | `git mv` 一次性 / 两阶段重叠 / 写迁移脚本                                 | **`git mv` 一次性 + 删旧**（保留 rename history）                     |

| tsconfig include  | `["src", "test"]` / `["src/**/*", "test/**/*"]` / 不动                    | **`include: ["src", "test"]`**（与现有 `["src"]` 风格一致）           |
| 任务范围          | 纯重组 / 重组+小补丁 / 重组+顺手补优化                                    | **重组 + 必要小补丁**（`.gitignore`、`tsconfig`、`vitest`、`README`） |
| 版本 bump         | bump 到 0.0.4 / 不 bump                                                   | **不 bump**（纯架构维护，无功能/契约变更）                            |
| 文档详细度        | 跟 v3 一样详尽 / 中等 / 简要                                              | **跟 v3 一样详尽**（用户决定）                                        |
| 跨测试文件 import | 提前设计 shared helper / 以现状为准                                       | **以现状为准**（已确认无跨测试 import，v4 不预设此场景）              |

**严格规则**：

- `test/<path>` 必须与 `src/<path>` 完全相同（包括大小写、连字符、点号）
- 客户端 `App.test.tsx` 直接位于 `client/test/`（与 `client/src/App.tsx` 平级）
- 客户端 `setup.ts` 位于 `client/test/setup.ts`（**不**放 `test/setup/setup.ts`）
- 删完 `src/test/` 后 `src/` 下不应残留任何空目录

### 5.6 错误处理

**典型失败模式**：

1. **vitest 报 "Cannot find module '@/foo'"**
   - 原因：`vitest.config.ts` 的 `resolve.alias` 没配或路径错
   - 修复：检查 `fileURLToPath(new URL('./src', import.meta.url))` 是否指向子项目根的 `src/`

2. **vitest 找不到测试文件**
   - 原因：`include` 没改或 `setupFiles` 路径仍指向 `src/test/`
   - 修复：grep `vitest.config.ts` 中的旧路径

3. **tsc 报 "Cannot find name" / "Cannot find module '@/foo'"**
   - 原因：tsconfig 的 `paths` 没生效（一般是 baseUrl 缺失或相对路径错）
   - 修复：确认 `tsconfig.json` 的 `paths: { "@/*": ["./src/*"] }` 和 `baseUrl: "."` 都在

4. **某个子项目 `pnpm test` 报"找不到 setup.ts"**
   - 原因：`client/vitest.config.ts` 的 `setupFiles: ['./test/setup.ts']` 路径错
   - 修复：检查 `git mv src/test/setup.ts test/setup.ts` 是否成功

5. **git diff 显示大量 add+delete 而非 rename**
   - 原因：用了 `mv` 而非 `git mv`
   - 影响：rename history 丢失；`git log --follow` 看不到
   - 修复：commit 前发现就用 `git mv` 重做；commit 后发现则需用 `git log --follow` 才能追历史

**任何一步失败** → 暂停，`git status` 确认状态，回滚或修补后继续；不要"先提交再修"。

### 5.7 测试策略

v4 自身**不写新测试**（纯架构维护，无功能变化）。验证手段是跑既有测试全绿：

1. **单元 / 组件测试**（`pnpm -r run test`）：32 个测试文件全部应通过
2. **typecheck**（`pnpm -r run typecheck`）：三个子项目全过
3. **lint**（`pnpm -r run lint`）：三个子项目全过
4. **git rename 验证**：
   ```bash
   git log --diff-filter=R --name-status HEAD~1..HEAD
   ```
   应显示 33 个 `R`（rename）行，**不应有 `A`（add）或 `D`（delete）对成对出现**指向同一文件
5. **src 净化验证**：
   ```bash
   find client/src gateway/src core/src -name "*.test.ts" -o -name "*.test.tsx"
   ```
   应输出**空**
6. **test 镜像验证**：
   ```bash
   diff <(find client/src -name "*.ts" -o -name "*.tsx" | sed 's|^client/src/||' | sort) \
        <(find client/test -name "*.ts" -o -name "*.tsx" | sed 's|^client/test/||' | sort -u)
   ```
   对 client / gateway / core 三个分别跑；测试文件对应的源文件必须在 `src/` 里存在

## 七、回滚与兼容

**回滚 v4 自身**：单个 commit / 单个 MR，revert 即回退。所有 33 个文件位置恢复、import 恢复、`tsconfig` / `vitest.config.ts` 恢复——v3 测试策略原样可用。

**对运行行为的影响**：**零**。被测代码未变，测试断言未变，测试运行命令未变。`pnpm -r run test` 跑出来仍是同一组测试、同一组通过率。

**对 CI 的影响**：**零**。`pnpm -r run typecheck` / `pnpm -r run test` / `pnpm -r run lint` 命令字面不变，CI 不需要任何调整。

**对调试体验的影响**：

- IDE（VS Code）跳转：测试文件 `import { ... } from '@/auth/store'` 跳转到 `src/auth/store.ts` 正常（tsconfig `paths` 已配）
- 测试覆盖率：报告生成位置（`<subproject>/coverage/`）未变
- 单文件运行：`pnpm -r --filter @my-ai/gateway exec vitest run test/auth/store.test.ts` 形式仍可用

**对开发流程的影响**：

- 写新测试时：放到 `test/<mirror-path>/<name>.test.ts(x)`，import 写 `'@/foo/bar'`
- 改源码位置时：测试文件**不需要**跟着改（alias 让位置解耦）
- Code review 时：习惯 `src/auth/store.ts` 的 reviewer 找 `test/auth/store.test.ts` 即可，路径一一对应

**v4 → v5 兼容性**：

- v5+ 继续使用 `@/*` alias（无破坏）
- v5+ 可选地引入 shared helper（`test/_helpers/`、`test/_mocks/`）——**v4 不预设**该结构
- v5+ 调整 `src/` 目录时（如把 `auth/` 拆成 `auth/store/` + `auth/middleware/`），只需把 `test/` 下对应子目录同步拆开，import 写 `@/auth/store/...` 即可
