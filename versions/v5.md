> 状态：🟡 待用户审阅

> 分支：`dev-20260609`
> 上一版本：[v4 — 测试目录统一与 path alias 化](./v4.md)

## 一、问题陈述

v4 合入后，client 端的 6 个组件（`App` / `MismatchBanner` / `PairBanner` / `PairDialog` / `Settings` / `ThemeToggle`）与 2 个错误处理函数（`friendlyNetworkError` / `friendlyApiError`）全部以中英混排的**字符串字面量**形式直接写在 JSX 中：

```tsx
// PairBanner.tsx
const kicker = isRepair ? '⚠ 重新配对' : '待配对';
const message = isRepair
  ? '上次的 clientKey 已失效，请发起新的配对请求。'
  : '尚未配对。请先完成网关配对。';

// PairDialog.tsx
const map: Record<number, string> = {
  400: '请求格式有误',
  401: '客户端标识无效或缺失',
  // ...
};
```

这种"硬编码在源代码里"的方式在 v3 之前不是问题——彼时 client 只服务中文用户。但 v3 合入后：

- **公开版的潜在受众不止中文用户**。README 写明 v3 起"私有模式 + pair key"是默认部署形态，但公开模式（`GATEWAY_PAIRING_PUBLIC=true`）理论上对任何能访问 gateway 的人开放——这些人未必读得懂中文文案。
- **PAIR_FAILED 等状态码返回是英文（"token_not_found" / "pair_pending"）**，被中文 UI 包住后用户能容忍；一旦加英文 UI，"token_not_found" 这种 snake_case 直接露出来很怪。
- **测试断言已与中文文案强耦合**。`PairBanner.test.tsx` 用 `screen.getByText(/未配对/)` 匹配；`Settings.test.tsx` 用 `screen.getByRole('button', { name: /测试/ })`。要做 i18n 必然要动这些断言——但断言改的"姿势"决定了未来再加语种时测试套件的稳定性。
- **错误码到中文文案的映射是 `Record<number, string>`，硬编码在 `friendlyApiError` 函数体内**。新增语言意味着复制整张 map 并本地化——重复且易漂移。
- **CLI 命令、placeholder、aria-label** 散落在 5 个 `src/components/` 下的组件里，没有集中点管理。

v5 的任务：**为 client 引入 i18n 架构，把现有 6 个组件（`App` + 5 个 `src/components/`）的中英混排文案抽取到资源文件，并实际交付简体中文（zh-CN）与英文（en）两套语言**。用户可在 UI 顶栏一键切换，状态可由 localStorage 持久化；资源文件与运行时解耦，后续新增语种只需在 `client/src/i18n/locales/<lng>/translation.json` 加一份即可，**无需**修改任何组件代码。

## 二、目标 / 非目标

### 目标

- **架构完整**：i18next + react-i18next 跑通，覆盖 6 个组件（含 `App`）的全部用户可见文案（含 aria-label / placeholder / 错误码 map / title / tooltip）。
- **两语种交付**：简体中文（zh-CN） + 英文（en），v5 即可在 UI 切换。
- **持久化**：用户切换语言后写 `localStorage['my-ai:lang']`；下次启动读这个键（优先于 `navigator.language`）。
- **降级链**：`localStorage` > `navigator.language`（主语种判定）> hard-code `'zh-CN'`。
- **测试不耦合到具体译文**：现有 5 个 `test/components/*.test.tsx` 断言改为 `screen.getByText(i18n.t('xxx'))`，key 稳定不变，译文体变不破坏测试。
- **CI 校验翻译完整性**：新增 `i18n-check` job，en.json 与 zh-CN.json 的 key 集合必须完全一致；任一文件 JSON 损坏或 value 非 string 即红。
- **品牌 / CLI / 版本号不翻译**：`my-ai`、`my-ai-gateway pair --token <token>`、`v3` / `CLIENT 0.0.3` / `GATEWAY v0.0.3` 这类"事实"信息不进 i18n 资源文件，原样保留。
- **零网络协议变更**：i18n 改动**全部**在 client 内部，gateway / core 不动；`X-Client-Key` / `/health` / `/pair` / `/v1/*` 等接口契约零变化。

### 非目标

- **不**支持 RTL（阿拉伯语 / 希伯来语）：v5 仅 zh-CN + en，两者都是 LTR；RTL 是 v6+ 评估。
- **不**做日期 / 数字本地化：footer 的 `2026/06/08` 与 `v0.0.3` 等仍是字面量；v5 不引入 i18next 的 Intl 配合。
- **不**做 Tauri 原生层 i18n：v5 仅 client 端 React 层；Tauri native menu / dialog title v6+ 评估。
- **不**做运行时语言检测增强：v5 仅 `localStorage` + `navigator.language`；IP 地理定位、Accept-Language 多 header 解析等 v6+。
- **不**做翻译协作流程：v5 翻译由代码 review 保证；Crowdin / Weblate 接入 v6+。
- **不**做 server-side 国际化：gateway / core 仍只输出英文 error code（`token_not_found` / `pair_pending` 等）；client 端在 `friendlyApiError` 内做码→本地化文案映射。
- **不**做动态加载资源 / lazy import：v5 静态 import 进 bundle，Tauri file:// 协议下无 CORS 顾虑，bundle +2 个 JSON 约 5KB 可接受。
- **不**改 client → gateway 的网络协议：`X-Client-Key`、`/health` 响应 schema、`{data, code, message}` 包装全部保持。
- **不**改 v1-v4 任何已有功能的运行行为：状态机、heartbeat、stronghold 持久化、CLI 子命令、DB schema 全部不变。

```
启动流程：
  main.tsx import './i18n'
    ├─ localStorage['my-ai:lang'] 命中 → lng = 该值
    ├─ 未命中 → navigator.language 主语种判定
    │   ├─ 'zh' → lng = 'zh-CN'
    │   ├─ 'en' → lng = 'en'
    │   └─ 其他 → lng = 'zh-CN'（fallback）
    └─ i18n.init({ resources, lng, fallbackLng: 'zh-CN' })

切换流程（用户点 LanguageSwitcher）：
  onClick → i18n.changeLanguage(next)
    ├─ react-i18next 自动 rerender 所有 t() 调用者
    ├─ localStorage.setItem('my-ai:lang', next)
    └─ 不派发事件、不刷新页面
```

### 5.2 i18n 初始化（`client/src/i18n/index.ts`）

```ts
// 客户端 i18n 初始化（v5）：静态 import 资源 + 检测链。
// 资源文件小（en + zh-CN 合计 < 5KB），无 HTTP 开销、无 CORS 顾虑（Tauri file:// 协议）。
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import en from './locales/en/translation.json';
import zhCN from './locales/zh-CN/translation.json';

const STORAGE_KEY = 'my-ai:lang';
const SUPPORTED = ['zh-CN', 'en'] as const;
type Supported = (typeof SUPPORTED)[number];

function detectLng(): Supported {
  // 1. localStorage
  try {
    const stored = localStorage.getItem(STORAGE_KEY);
    if (stored && (SUPPORTED as readonly string[]).includes(stored)) {
      return stored as Supported;
    }
  } catch {
    // 隐私模式 / localStorage 不可用 → 降级
  }
  // 2. navigator.language 主语种
  const primary = typeof navigator !== 'undefined' ? (navigator.language.split('-')[0] ?? '') : '';
  if (primary === 'zh') {
    return 'zh-CN';
  }
  if (primary === 'en') {
    return 'en';
  }
  // 3. fallback
  return 'zh-CN';
}

i18n
  .use(initReactI18next)
  .init({
    resources: {
      en: { translation: en },
      'zh-CN': { translation: zhCN },
    },
    lng: detectLng(),
    fallbackLng: 'zh-CN',
    interpolation: {
      escapeValue: false, // React 已防 XSS，i18next 不必再 escape
    },
    returnNull: false, // 缺 key 时返回 fallbackLng 串，不返回空串
  })
  .catch((e: unknown) => {
    // i18n init 失败时只 log，不阻塞应用启动
    // （v5 资源是静态 import，理论上 init 不会失败；catch 仅兜底）
    console.error('i18n init failed:', e);
  });

export { SUPPORTED };
export default i18n;
```

**`main.tsx` 集成**：

```tsx
// v5: 触发 i18n 初始化（side-effect import）
import './i18n/index.js';
// ... 既有 ReactDOM.createRoot
```

**key 命名规则**（review 时核对）：

- 一级 = feature / domain：`app` / `settings` / `pair` / `mismatch` / `errors` / `theme` / `lang`
- 二级 = 子模块：`pair.banner` / `pair.dialog` / `errors.api`
- 状态 / 类型用大写保留：`PAIRING` / `HEALTHY` / `MISMATCH` / `PAIR_FAILED`
- 插值占位 `{{var}}`（i18next 默认语法，不引 `react-i18next` 之外的库）
- 状态码作 key（`errors.api.400`）；不引 `i18next-icu` 等

**保留不翻译的常量**（写在组件内，不进 i18n 资源）：

- `my-ai`（品牌）
- `CLIENT 0.0.3` / `v0.0.3` / `v3`（版本号字面量）

### 5.5 LanguageSwitcher（`i18n/switch.tsx`）

```tsx
// 顶栏语言切换按钮（v5）：zh-CN ↔ en。点轮换 + 写 localStorage。
// 与 ThemeToggle 同级（header 内右侧）。2 个语种用按钮最简，不做下拉。
import { useTranslation } from 'react-i18next';
import { SUPPORTED } from './index.js';

const NEXT: Record<string, string> = { 'zh-CN': 'en', en: 'zh-CN' };

export function LanguageSwitcher() {
  const { i18n, t } = useTranslation();
  const current = i18n.language;
  const next = NEXT[current] ?? 'en';
  const nextLabel = t(`lang.${next}`);
  const currentLabel = t(`lang.${current}`);

  const handleClick = () => {
    void i18n.changeLanguage(next).then(() => {
      try {
        localStorage.setItem('my-ai:lang', next);
      } catch {
        // 隐私模式 / localStorage 不可用 → 静默忽略（会话期内有效即可）
      }
    });
  };

  return (
    <button
      type="button"
      className="lang-toggle"
      data-current={current}
      onClick={handleClick}
      aria-label={t('lang.aria', { current: currentLabel })}
      title={`${currentLabel} → ${nextLabel}`}
    >
      <span className="dot" aria-hidden="true" />
      <span>{currentLabel}</span>
    </button>
  );
}

export { SUPPORTED };
```

**形态**：

- 视觉与 ThemeToggle 对齐（圆点 + 文字）
- 文字显示**当前**语言代码（`中` / `EN`）
- 点 → 切到另一语种 + 写 localStorage
- 不派发自定义事件、不刷新页面
- `aria-label` / `title` 走 i18n

**App.tsx 集成**：

```tsx
<header className="app-meta">
  <span className="brand">{t('app.brand')}</span>
  <ThemeToggle />
  <LanguageSwitcher /> {/* v5 新增 */}
</header>
```

### 5.6 错误码本地化（PairDialog.tsx 重构）

**重构前**（v4 现状）：

```ts
function friendlyApiError(e: unknown, prefix: string): string {
  if (!(e instanceof ApiError)) {
    return friendlyNetworkError(e, prefix);
  }
  const map: Record<number, string> = {
    400: '请求格式有误',
    401: '客户端标识无效或缺失',
    403: '无权限访问',
    404: '资源不存在（确认码可能已过期）',
    500: '网关内部错误',
    502: '上游服务不可用',
  };
  const zh = map[e.code];
  if (zh) {
    return `${prefix}：${zh}`;
  }
  // ...
}
```

**重构后**（v5）：

```ts
function friendlyApiError(e: unknown, prefix: string): string {
  if (!(e instanceof ApiError)) {
    return friendlyNetworkError(e, prefix);
  }
  const t = useT(); // 假设组件内 useTranslation 拿到的 t 函数
  const key = `errors.api.${e.code}`;
  // i18next 不存在的 key 会返回 key 字面量（returnNull: false 时），
  // 但我们 fallbackLng='zh-CN' 保证至少回退到中文
  const translated = t(key, { defaultValue: '' });
  if (translated) {
    return `${prefix}：${translated}`;
  }
  // 兜底：message 可能为 undefined（网关漏字段）
  const raw = (e.message ?? '').toString();
  if (raw && raw !== 'undefined' && raw !== 'null') {
    return `${prefix}：${raw}`;
  }
  return `${prefix}：错误码 ${e.code}`;
}
```

**关键差异**：

- 错误码 → 文案映射从函数体内的字面量 Record 改为资源文件 key
- 400-502 共 6 个码的文案进 `errors.api.<code>`
- 未列出的码走 `defaultValue: ''` + 兜底 message
- 翻译变更**不改** PairDialog.tsx

`friendlyNetworkError` 同步重构：错误文案从字面量改查 `errors.network.{fetchFail, notMyAi, unknown}`。

### 5.8 持久化与启动降级

**localStorage 键**：`my-ai:lang`，值 `'zh-CN'` 或 `'en'`。

**写入时机**：

- 用户点 LanguageSwitcher → `localStorage.setItem('my-ai:lang', next)`
- 不在 init 阶段写（init 只读）

**读取时机**：

- `i18n/index.ts` 模块加载时（被 `main.tsx` 引用即触发）
- 优先级最高（即使 navigator.language 是 en，localStorage 存了 zh-CN 仍用 zh-CN）

**降级链**：

```
1. localStorage['my-ai:lang']
   ├─ 命中 + 在 SUPPORTED 列表 → 用
   └─ 未命中 / 不在 SUPPORTED → 继续
2. navigator.language 主语种
   ├─ 'zh' → 'zh-CN'
   ├─ 'en' → 'en'
   └─ 其他 → 'zh-CN'（默认）
3. hard-code 'zh-CN'
```

**异常处理**：

- localStorage 读失败（隐私模式 / 不可用）→ 静默 catch，继续 navigator 步骤
- navigator 不可用（SSR 边界）→ 返回 `'zh-CN'`
- i18n.init 失败（理论不会，资源静态 import）→ `console.error` + 应用仍启动（fallbackLng='zh-CN' 保证 t() 至少返回 key）

**跨进程一致性**：

- localStorage 是 per-origin 存储；Tauri 桌面端每个 instance 一个 origin，跨实例不共享（v5 不解决）
- Tauri 移动端（v6+）会改为 stronghold；v5 仅桌面

**`scripts/check-i18n.mjs`**：

```js
// v5: i18n 资源完整性校验。
// 规则：
//  1. en/translation.json 与 zh-CN/translation.json 都是合法 JSON
//  2. 两个文件 key 集合完全一致
//  3. 任何 value 不是 string 即 fail
// 失败码非 0 → CI 红。
import { readFileSync } from 'node:fs';
import { resolve } from 'node:path';

const LOCALES_DIR = resolve('client/src/i18n/locales');
const LOCALES = ['en', 'zh-CN'];

function loadJson(path) {
  const text = readFileSync(path, 'utf8');
  try {
    return JSON.parse(text);
  } catch (e) {
    console.error(`✖ JSON 解析失败: ${path}\n  ${e.message}`);
    process.exit(1);
  }
}

function flattenKeys(obj, prefix = '') {
  const keys = [];
  for (const [k, v] of Object.entries(obj)) {
    const path = prefix ? `${prefix}.${k}` : k;
    if (v !== null && typeof v === 'object' && !Array.isArray(v)) {
      keys.push(...flattenKeys(v, path));
    } else {
      keys.push({ path, value: v });
    }
  }
  return keys;
}

function checkValueIsString(entries, file) {
  for (const { path, value } of entries) {
    if (typeof value !== 'string') {
      console.error(`✖ ${file}: key "${path}" 的 value 不是 string（实际类型 ${typeof value}）`);
      return false;
    }
  }
  return true;
}

const flat = {};
for (const lng of LOCALES) {
  const path = resolve(LOCALES_DIR, lng, 'translation.json');
  flat[lng] = loadJson(path);
  const entries = flattenKeys(flat[lng]);
  if (!checkValueIsString(entries, `${lng}/translation.json`)) {
    process.exit(1);
  }
}

const keysByLng = {};
for (const lng of LOCALES) {
  keysByLng[lng] = new Set(flattenKeys(flat[lng]).map(e => e.path));
}

let failed = false;
for (const lng of LOCALES) {
  for (const other of LOCALES) {
    if (lng === other) continue;
    const missing = [...keysByLng[other]].filter(k => !keysByLng[lng].has(k));
    const extra = [...keysByLng[lng]].filter(k => !keysByLng[other].has(k));
    if (missing.length > 0) {
      console.error(
        `✖ ${lng}/translation.json 缺少 keys（相对 ${other}）:\n  ${missing.join('\n  ')}`,
      );
      failed = true;
    }
    if (extra.length > 0) {
      console.error(
        `✖ ${lng}/translation.json 多出 keys（相对 ${other}）:\n  ${extra.join('\n  ')}`,
      );
      failed = true;
    }
  }
}

if (failed) {
  process.exit(1);
}

console.log(`✓ i18n check passed: ${LOCALES.length} locales, key 集合一致`);
```

- `quality` job 顺序：`typecheck` / `lint` / `test` / `dedupe` / `secret scan` / `compat-matrix-check`
- **不**把 `i18n-check` 塞进 quality——按 v2 经验，独立的 job 失败原因清晰
- `i18n-check` 跑在 `quality` 之后、`build-*` 之前

### 5.11 错误处理

| 阶段    | 错误                                   | 处理                                                              |
| ------- | -------------------------------------- | ----------------------------------------------------------------- |
| Build   | 资源文件 JSON 损坏                     | Vite 编译失败；CI `i18n-check` 也兜底再扫一遍                     |
| Build   | en.json 与 zh-CN.json key 集合不一致   | CI 红（§5.10）                                                    |
| Build   | 任一 value 不是 string                 | CI 红                                                             |
| Build   | 任一 value 是 React 元素（写错）       | 编译期 TypeScript 不通过                                          |
| Startup | localStorage 读失败（隐私模式）        | 静默 catch；走 navigator 检测                                     |
| Startup | navigator 不可用（SSR 边界）           | 返回 `'zh-CN'`                                                    |
| Startup | i18n.init 失败（理论不会）             | `console.error` + 应用仍启动（fallbackLng='zh-CN' 保证 t() 不崩） |
| Runtime | 切语种时新语种资源缺 key               | i18next fallbackLng='zh-CN' 自动降级                              |
| Runtime | 切语种时 localStorage 写失败           | 静默 catch；会话期有效即可                                        |
| Runtime | `t(key, { var })` 缺 `{{var}}` 插值    | i18next 保留占位符字面量（不抛错）                                |
| Runtime | 资源文件运行时被改（理论上 Vite 不会） | dev 模式 hot reload 触发；prod 不可改                             |
| Test    | 组件测试未引入 i18n                    | `i18n.t` 抛 "i18next not initialized"；setup.ts 强制 import       |

**核心原则**（与 v2/v3 一致）：

- **build / startup loud fail**：编译期错误、CI 校验失败 → 显式红；不在生产期才暴露
- **runtime soft fail**：运行期缺资源、localStorage 失败、navigator 不可用 → 走降级路径，不中断用户

## 七、回滚与兼容

**回滚 v5 自身**：单 commit / 单 MR，revert 即回退。所有 v5 引入的文件随 revert 一起消失；5 个组件测试的 `i18n.t(...)` 改回 v4 形式的字面量断言（5 个测试文件），回到 v4 状态。

**对 v4 老安装的兼容**：

- v4 client / v4 gateway：完全无感，v5 改动**全部在 client 内部**
- v5 client 调 v4 gateway：网络协议、API、鉴权流程均不变（i18n 不影响）
- v5 client 调 v5 gateway（合入后）：v5 gateway 未改，行为与 v4 gateway 一致

**对调试体验的影响**：

- IDE 跳转：`import i18n from '@/i18n/index.js'` 跳转到 `src/i18n/index.ts` 正常（v4 已配 `@/*` alias）
- 资源文件 IDE 跳转：JSON 键 `t('pair.banner.needPair.kicker')` 在 VS Code 可装 i18n-ally 插件做 key ↔ value 跳转（v5 不强制装）
- 调试切语种：DevTools console 跑 `i18n.changeLanguage('en')` 立即生效
- localStorage 检查：`localStorage.getItem('my-ai:lang')` 即可

**对开发流程的影响**：

- 写新组件时：用 `useTranslation()` 拿 `t` 函数，**不要**写硬编码中文字符串
- 改组件文案时：先改 `zh-CN/translation.json` 与 `en/translation.json` 两个文件，CI `i18n-check` 兜底
- 加新语种时：
  1. 在 `client/src/i18n/locales/<新语种>/translation.json` 加资源文件
  2. 在 `client/src/i18n/index.ts` 的 `SUPPORTED` 数组追加
  3. 在 `<LanguageSwitcher />` 的 `NEXT` map 与 `lang.<code>` 资源中加映射
  4. CI `i18n-check` 校验新语种 key 集合与其他语种一致
  5. 现有 6 个组件（`App` + 5 个 `src/components/`）代码不变
- Code review 时：习惯 `src/components/Settings.tsx` 的 reviewer 看文案来源时跳到 `src/i18n/locales/zh-CN/translation.json` 即可

**v5 → v6 兼容性**：

- 资源文件结构稳定，v6 加新语种无需改组件代码
- `i18n/index.ts` 的 `SUPPORTED` 数组追加即可
- 现有 6 个组件（`App` + 5 个 `src/components/`）代码不变
- compat range 维持 `>=0.0.3 <0.1.0` 不变

## 八、待处理（移交给 v6 / 后续）

- **更多语种**：日 / 韩 / 法 / 德 / 俄。v5 仅 zh-CN + en，资源文件结构 + i18n.init 已支持扩展（v6+ 评估）
- **RTL 支持**：阿拉伯语 / 希伯来语。v5 不涉及，v6+ 评估 CSS `direction: rtl` 与 `logical properties`（CSS Logical Properties 已广泛支持）
- **Tauri 端的 i18n**：menu / dialog title / 系统通知。v5 仅 client 端 React 层；Tauri native i18n v6+ 评估（涉及 `tauri.conf.json` 的 `bundle.i18n` 字段）
- **日期 / 数字本地化**：footer `2026/06/08` 与版本号 `v0.0.3` 当前为字面量；v5 不引入 i18next 的 Intl 配合，v6+ 评估
- **运行时语言检测增强**：v5 仅 `localStorage` + `navigator.language`；IP 地理定位、Accept-Language 多 header 解析 v6+ 评估
- **翻译协作流程**：v5 翻译由代码 review 保证；Crowdin / Weblate / PO 文件接入 v6+ 评估

- **gateway / core 端的 i18n**：v5 不动；如未来 gateway 需要返回多语言错误码（接受 `Accept-Language` header），v6+ 评估（涉及 `/health` 与 `/pair` 响应 schema 变更）
- **测试覆盖 en 文案**：v5 钉死测试为 zh-CN，仅断言 zh-CN 译文存在；v6+ 评估"两语种都跑一遍组件测试"（matrix 跑，CI 时间会变长）

## 九、附录：brainstorming 决策记录

| #   | 决策                                                    | 原因                                                                                                                                      |
| --- | ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | i18next + react-i18next                                 | 业界事实标准；React 生态最成熟；Tauri + React 18 兼容；hooks API (`useTranslation`) 与现有组件风格一致                                    |
| 2   | 静态 import 资源进 bundle                               | 无 HTTP 开销；Tauri file:// 协议无 CORS 顾虑；bundle +2 JSON 约 5KB 可接受                                                                |
| 3   | 点分嵌套 key 路径                                       | Feature 边界清晰；JSON 体积小；调用 `t('pair.banner.needPair.kicker')` 可读性高                                                           |
| 4   | localStorage 存 `my-ai:lang`                            | 与 ThemeToggle 风格一致（同样用 `my-ai:` 前缀）；语言非敏感信息，不需 stronghold                                                          |
| 5   | 切换器在 Header 顶栏（ThemeToggle 旁）                  | 全局可见；2 个语种用按钮最简；用户偏好"控件偏好"集中放顶栏                                                                                |
| 6   | 切换器是按钮（zh↔en）                                   | 2 个语种用按钮切换最简；下拉需要 2 次点击；radio 占空间                                                                                   |
| 7   | 测试断言用 `i18n.t(key)`                                | key 稳定不变，译文体变不破坏测试；测试与具体语言解耦                                                                                      |
| 8   | 测试环境钉死 zh-CN                                      | 避免依赖 `navigator.language` 在 CI 中不稳定；测试断言直接用 i18n.t(key) 返回值                                                           |
| 9   | 资源文件位置 `src/i18n/locales/{en,zh-CN}/`             | 与 `src/lib/` `src/compat/` 风格一致；嵌套目录隔离各语种                                                                                  |
| 10  | 翻译完整性 CI 校验                                      | 缺 key 是高频错误；脚本轻量、CI 红即拦截；与 v2 `compat-matrix-check` 同风格                                                              |
| 11  | client version 0.0.3 → 0.0.4                            | 仅 client 改代码；v2 规则"改某端代码 → 该端 version 必升"；v4 没收这个改动是因为 v4 是"纯架构维护无功能"，v5 是"加新功能（i18n 切换 UI）" |
| 12  | compat range 不变                                       | 0.0.4 client 仍兼容 0.0.3 gateway，向下兼容；新功能纯前端                                                                                 |
| 13  | 品牌 / CLI 命令 / 版本号 不进 i18n 资源                 | 这些是"事实"不是"翻译"；进 i18n 反而增加维护成本；CLI 命令必须原样（用户在终端跑）                                                        |
| 14  | a11y 属性值（role / aria-hidden / data-state）不进 i18n | 这些是协议值（a11y tree、selector），不是用户可见文案；不同语种应保持相同值以保证 a11y 工具可解析                                         |
| 15  | 切语种时仅 `i18n.changeLanguage` + localStorage 写入    | react-i18next 自动 rerender 所有 `t()` 调用者；不派发事件、不刷新页面；最小副作用                                                         |
| 16  | 缺 key 走 `fallbackLng='zh-CN'`                         | 不抛错；不显示 key 字面量；保证 UI 永远有文案；CI 兜底翻译完整性                                                                          |
| 17  | 错误码本地化从函数内 Record 改为资源文件                | 错误码 → 文案映射是 i18n 的核心 use case；记录在 PairDialog 函数内意味着新增语言需复制整张 map                                            |
| 18  | STATUS_LABEL 改为运行时 `t('settings.status.<STATUS>')` | 4 个状态文案从 module-level Record 改为运行时调用；状态名作为 key 末段                                                                    |
| 19  | `i18n-check` 是独立 job                                 | 失败原因清晰；不塞进 quality job；与 v2 `version-bump-check` 风格一致                                                                     |
| 20  | localStorage 写入失败静默忽略                           | 隐私模式 / 不可用是少数场景；会话期内 i18n 仍生效；用户体验优先                                                                           |
