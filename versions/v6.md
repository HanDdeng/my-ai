# v6 — 多 agent 协作架构（虚拟办公室）

> 状态：🟡 设计文档定调，待用户审阅
> 日期：2026-06-10
> 分支：`dev-20260610`
> 上一版本：[v5 — 客户端国际化（i18n）](./v5.md)

## 一、问题陈述

v5 合入后（commit `29971b7`）my-ai 现状是：

- **三端骨架**（v1 拉起 / v2 握手 / v3 远程配对鉴权 / v4 测试目录与 path alias / v5 i18n）已经定型。
- **core 端** 已有 `AgentRegistry` + `EchoAgent`（v1 起的"agent 核心"）+ `/v1/agents` 列表端点 + `/v1/chat` 单轮同步路由。
- **gateway 端** 透传 `/v1/agents` 到 core（v3 鉴权化）。
- **client 端** 现状是"配对 / 握手 / 设置 + i18n"，**完全没有 agent 列表 / chat 消息流 / 任何 agent 相关的 UI**。

README "后续计划"里列了两件未做之事：

> - [ ] client 多 agent 切换 UI
> - [ ] 多 agent 协作与配置中心

v5 合入后，client 端 "i18n 已经能 zh-CN / en 切换"，但 README 那两件事仍未落地，client 端看不到任何 agent、不能调任何 chat 路由。core 端那个 `EchoAgent` 仍是"在 `buildServer()` 里 `registry.register(new EchoAgent(llm))` 硬编码"——这是 v1 的设计，**没有 CRUD 入口、没有持久化、不能被用户配置**。

v6 的任务：**为 my-ai 引入"多 agent 协作架构"的定调设计**——三端职责切分、数据模型、CRUD API、客户端"虚拟办公室"UI 形态、v5 现状如何过渡、v7+ 留什么口子。

**v6 仅是设计文档**——本版本**不**含实现计划、不**规划** v6 实现顺序。v6 范围里的具体三端代码实现按"agent × session × UI"维度拆给 v7+ 各自承接。

## 二、目标 / 非目标

### 目标

- **三端职责清晰**：
  - **core 端**：agent 配置 / session / message 持久化的**权威源**（新增 SQLite 三张表）；registry 从"硬编码"改为"DB 驱动"；新增 6 个 CRUD 端点 + 同步 chat 路由。
  - **gateway 端**：纯透传 + v3 鉴权；不存 agent 数据。
  - **client 端**：保留 v5 现状 PairBanner / PairDialog / Settings 不动；**PAIRED 状态新增"虚拟办公室"画布**——agent 角色卡片 + 点击弹窗 chat。
- **完整 CRUD**：用户可在 client 端对 agent 增 / 改 / 删 / 查 / 列表 / 切 session / 看历史 / 发消息。
- **session 持久化**：core 端按 session 持久化 messages 表，agent × session 1:1 绑定（切 agent = 开新 session）。
- **LLM 接口预留**：v6 仅实现 `mock` provider；`openai-compatible` / `anthropic` / `ollama` 等 case 注册可、调 chat 走 501 `not_implemented`；`createLLMClient` 重构为 `Map<provider, factory>` 可插拔。
- **v1 EchoAgent 硬编码完全删**：`core/src/agent/echo.ts` + `buildServer()` 注入 + `test/agent/registry.test.ts` 硬编码 case 全删；v6 起所有 agent 100% 从 DB 读。
- **首次启动不预置**：core 端首次启动 agent 表为空；client 主屏显示"尚无 agent，请新建一个"。
- **i18n 集成**：v6 新增 UI 文案（虚拟办公室标题 / 空状态 / 按钮 / 弹窗标题 / placeholder / "user" "agent" 标签）全进 `client/src/i18n/locales/{en,zh-CN}/translation.json`；agent.name / agent.description 仍是**用户数据**，不进 i18n。
- **v5 兼容**：v5 client / v5 gateway / v5 core 的现有路径与流程全部保留；v6 增的 API 对 v5 透明。
- **零网络协议破坏**：v6 改的是"加新端点" + "core 内部数据流"；`X-Client-Key` 鉴权、响应 `{ok, data, code, message}` 包装、版本握手 compat 全部不变。

### 非目标

- **不**实现具体 LLM provider：v6 阶段只有 `mock` 可用；`openai-compatible` / `anthropic` / `ollama` 仅在 `agents.llm_provider` CHECK 约束中预留，调 chat 走 501 `not_implemented`。
- **不**做流式 chat 响应：v6 仍为同步（沿用 v1 chat 路由语义）。
- **不**做 agent 软删除 / archive / 标签 / 分组 / 搜索：v6 删除即真删（CASCADE）；上述能力 v7+ 评估。
- **不**做 pair 弹窗化：v5 PairBanner / PairDialog / Settings 在 v6 阶段**完全保留**；pair 弹窗化为 v7+ 评估。
- **不**做 agent 启用 / 停用开关：v6 注册即活跃；v7+ 评估。
- **不**做多 agent 弹窗并存：v6 一次只弹一个 chat 弹窗；v7+ 评估多窗口。
- **不**做 plan 写 / 实现顺序：v6 仅出设计文档；v6 范围里具体三端代码实施按维度拆给 v7+。
- **不**改 compat 矩阵范围：v6 涉及 core / gateway / client 三端代码改动，**预留 v6+ 各端 version bump 由 v7 实际写代码时定**（v6 文档不锁 bump 数字）。
- **不**改 v1-v5 任何已有功能的行为：状态机 / heartbeat / stronghold / i18n / compat 全部保持。

### 5.2 数据模型（core 端 SQLite）

```sql
-- agents 表：用户 CRUD 的对象
CREATE TABLE agents (
  id            TEXT PRIMARY KEY,                  -- client 生成 UUID v4
  name          TEXT NOT NULL,                     -- 用户可改；client 端查重提示
  description   TEXT NOT NULL DEFAULT '',
  capabilities  TEXT NOT NULL DEFAULT '[]',        -- JSON 数组（如 ["chat"] / ["chat","tool"]）
  llm_provider  TEXT NOT NULL DEFAULT 'mock',      -- v6 唯一可用值
  llm_config    TEXT NOT NULL DEFAULT '{}',        -- JSON map（v6 不含 api_key）
  system_prompt TEXT NOT NULL DEFAULT '',
  created_at    TEXT NOT NULL,                     -- ISO8601
  updated_at    TEXT NOT NULL,
  CHECK (llm_provider IN ('mock', 'openai-compatible', 'anthropic', 'ollama'))
  -- v6 阶段只有 'mock' 真正可用；其他 case 注册可、调 chat 走 501 not_implemented
);
```

```sql
-- sessions 表：每个 session 绑一个 agent
CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,                    -- client 生成 UUID v4
  agent_id    TEXT NOT NULL,                       -- 不可改；切 agent = 开新 session
  client_key  TEXT NOT NULL,                       -- 关联 gateway 端 clientKey（v6 存明文）
  title       TEXT NOT NULL DEFAULT '',            -- v6 留空；v7+ 评估"取首条消息前 N 字"
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL,
  FOREIGN KEY (agent_id) REFERENCES agents(id) ON DELETE CASCADE
);
```

```sql
-- messages 表：会话上下文窗口
CREATE TABLE messages (
  id          TEXT PRIMARY KEY,                    -- client 生成 UUID v4（保证顺序）
  session_id  TEXT NOT NULL,
  role        TEXT NOT NULL,                       -- user / assistant / system
  content     TEXT NOT NULL,
  created_at  TEXT NOT NULL,
  FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);
CREATE INDEX idx_messages_session_id ON messages(session_id);
```

**约束**：

- **agent `id`**：client 生成；删除 agent → CASCADE 删 sessions / messages。
- **session `id`**：client 生成；`agent_id` 字段**不可改**（切 agent = 开新 session）。
- **session `client_key`**：明文存；v6 不做脱敏（v3 鉴权已上，DB 仅本机 core 访问）。
- **message 顺序**：用 `id`（UUID v4 字典序）保证，**不**依赖 `created_at`。
- **LLM provider 校验**：`agents.llm_provider` CHECK 约束限定四选一（v6 唯一可用为 `mock`）。
- **真删**：v6 不做软删除 / archive / 启用-停用开关。
- **id 不复用**：删 agent 不复用 id（客户端不感知）。

### 5.3 API（gateway 透传 + core 真做；保留 v3 鉴权）

| Method | Path                       | 用途                                  | 鉴权 |
| ------ | -------------------------- | ------------------------------------- | ---- |
| GET    | /v1/agents                 | 列 agent                              | ✅   |
| POST   | /v1/agents                 | 新建 agent                            | ✅   |
| GET    | /v1/agents/{id}            | 取单个 agent                          | ✅   |
| PATCH  | /v1/agents/{id}            | 改 agent                              | ✅   |
| DELETE | /v1/agents/{id}            | 删 agent（CASCADE sessions/messages） | ✅   |
| POST   | /v1/sessions               | 开新 session（body: `{agentId}`）     | ✅   |
| GET    | /v1/sessions/{id}          | 取单个 session                        | ✅   |
| DELETE | /v1/sessions/{id}          | 删 session（CASCADE messages）        | ✅   |
| GET    | /v1/sessions/{id}/messages | 列历史消息                            | ✅   |
| POST   | /v1/sessions/{id}/messages | 发一条消息（同步 chat，取代 v1 chat） | ✅   |

**保留路径**：

- `GET /v1/agents`：v1 即有；v6 改为查 DB（v1 走 registry 内存）。
- `POST /v1/chat`：v1 即有；v6 保留但 v6 文档标注 "v7 评估删除"（v6 客户端不再调）。

**新路径**（v6 新增）：除 GET /v1/agents 外 9 个路径。

**错误码（沿用 v1 风格）**：

| 码  | 含义            | 触发                                                       |
| --- | --------------- | ---------------------------------------------------------- |
| 0   | ok              | 正常                                                       |
| 400 | invalid_body    | zod 校验失败                                               |
| 401 | unauthorized    | X-Client-Key 缺失 / 无效（v3 鉴权）                        |
| 404 | agent_not_found | agentId / sessionId 不存在                                 |
| 500 | internal_error  | 兜底                                                       |
| 501 | not_implemented | agent.llm_provider 非 mock（v6 阶段尚未实现具体 provider） |
| 502 | upstream_error  | core 不可达 / 响应异常                                     |

### 5.4 核心数据流：发一条消息

```
client (虚拟办公室，agent 弹窗)
  │
  │ POST /v1/sessions/{sid}/messages
  │ header: X-Client-Key: <clientKey>
  │ body: { content: "..." }
  │
  ▼
gateway (v3 鉴权 + 透传)
  │
  │ 内部 HTTP（新增 X-Internal-Client-Key header，把 clientKey 透传给 core）
  │  ——v1 阶段 gateway 透传时不带 clientKey；v6 起为支持 sessions.client_key 字段而新增
  │
  ▼
core:
  1. 鉴权（X-Client-Key 校验由 gateway 完成；core 仅信任 gateway 内部 header `X-Internal-Client-Key`，把 clientKey 写 `sessions.client_key` 字段）
  2. 查 session → 拿 agent_id
  3. 查 agent → 拿 llm_provider / llm_config / system_prompt
  4. 查 messages 历史（按 id 排序）→ 拼上下文
  5. createLLMClient(agent.llm_provider, agent.llm_config).chat({ systemPrompt, messages: [...history, new] })
     ├─ 'mock' → MockLLMClient（v1 即有，行为不变）
     └─ 其他 → 501 not_implemented
  6. 拿到 assistant reply
  7. DB insert user message + assistant message
  8. 同步返回 { userMessage, assistantMessage }
```

**关键路径**（详细）：

```ts
// 1. client 端选 agent（用户点卡片），生成 sessionId / userMessageId
const sessionId = randomUUID();
const userMessageId = randomUUID();

// 2. client 调 POST /v1/sessions 开新 session
//    body: { id: sessionId, agentId }
//    → core 真做 INSERT sessions 行；返回 session 对象
const session = await apiFetch(`/v1/sessions`, {
  method: 'POST',
  body: { id: sessionId, agentId: '...' },
  clientKey,
});

// 3. client 调 POST /v1/sessions/{sid}/messages 发消息
//    body: { id: userMessageId, content: "..." }
//    → core 真做 chat；返回 { userMessage, assistantMessage }
const { userMessage, assistantMessage } = await apiFetch(`/v1/sessions/${sessionId}/messages`, {
  method: 'POST',
  body: { id: userMessageId, content: '...' },
  clientKey,
});
```

### 5.6 createLLMClient 重构

**v1 现状**（`core/src/llm/index.ts`）：

```ts
export function createLLMClient(cfg: Config): LLMClient {
  switch (cfg.LLM_PROVIDER) {
    case 'mock':
      return new MockLLMClient();
    case 'openai-compatible':
      throw new Error('not implemented');
  }
}
```

**v6 重构**（伪代码）：

```ts
type Provider = 'mock' | 'openai-compatible' | 'anthropic' | 'ollama';
type Factory = (llmConfig: Record<string, unknown>) => LLMClient;

const factories = new Map<Provider, Factory>([
  ['mock', cfg => new MockLLMClient()],
  // v6 阶段不注册其他 provider；v7+ 加
  // ['openai-compatible', (cfg) => new OpenAICompatibleClient(cfg)],
]);

export function createLLMClient(provider: Provider, llmConfig: Record<string, unknown>): LLMClient {
  const factory = factories.get(provider);
  if (!factory) {
    throw new LLMNotImplementedError(provider);
  }
  return factory(llmConfig);
}

// core 端 chat 路由捕获 LLMNotImplementedError → 501 not_implemented
```

**`Config.LLM_PROVIDER` 退化为"全局默认"**：

- core 启动时如有 agent 缺 llm_provider（理论上 CHECK 约束保证不会），用 `Config.LLM_PROVIDER` 兜底
- 实际上 v6 CHECK 约束 + client 端校验保证 agent.llm_provider 必填且合法
- v7+ 评估：是否还保留 `Config.LLM_PROVIDER`

**chat 路由的 agent 实例化（lazy）**：

```ts
// 伪代码：v6 core chat 路由
//   userMessageId 来自 client 端 POST body；assistant message id 由 core 生成。
async function handleMessage(sessionId: string, userMessageId: string, content: string) {
  const session = db.sessions.get(sessionId);
  if (!session) throw new SessionNotFoundError(sessionId);

  const agent = db.agents.get(session.agent_id);
  if (!agent) throw new AgentNotFoundError(session.agent_id);

  // lazy 实例化（v6 阶段所有 agent 都用 EchoAgent 跑 mock；v7+ 评估按 llm_provider 分发）
  const instance = new EchoAgent(createLLMClient(agent.llm_provider, JSON.parse(agent.llm_config)));

  const result = await instance.run({
    agentId: agent.id,
    sessionId: session.id,
    message: { role: 'user', content },
    // v6 不传 history；v6 阶段 history 仅作"展示"，不影响 LLM 上下文窗口
  });

  // 写 user + assistant 两条 messages；id 各异
  db.messages.insert({ id: userMessageId, session_id: sessionId, role: 'user', content });
  const assistantMessageId = randomUUID();
  db.messages.insert({
    id: assistantMessageId,
    session_id: sessionId,
    role: 'assistant',
    content: result.reply.content,
  });

  return {
    userMessage: { id: userMessageId, role: 'user', content },
    assistantMessage: {
      id: assistantMessageId,
      role: 'assistant',
      content: result.reply.content,
      finishedAt: result.finishedAt,
    },
  };
}
```

**关键说明**：

- v6 阶段 `EchoAgent.run()` **不**用 `history` 字段；history 仅作 UI 展示。
- v7+ 评估：让 `agent.run()` 接收 history 参数、context window 滑动等。
- agent 实例不缓存（v6 简化）；v7+ 评估按 `agent.id` LRU 缓存。

### 5.8 v5 → v6 兼容矩阵

| 维度                            | v5 client 调 v6 core    | v6 client 调 v5 core           | v6 client 调 v6 core                                       |
| ------------------------------- | ----------------------- | ------------------------------ | ---------------------------------------------------------- |
| GET /v1/agents                  | 拿空列表（DB 空）       | 拿 EchoAgent 列表（v5 硬编码） | 拿 DB 列表                                                 |
| POST /v1/agents                 | 401（v5 core 无此路径） | 404                            | 新建成功                                                   |
| POST /v1/chat                   | 正常工作                | 正常工作                       | 正常工作（但 v6 client 调新路径，v1 chat 仅 v5 client 用） |
| POST /v1/sessions/{id}/messages | 404                     | 404                            | 同步 chat                                                  |

**结论**：

- **v5 client** 调 v6 core / v6 gateway：行为兼容（v5 client 调 GET /v1/agents 拿空列表 = "未建 agent"，不影响 v5 client 现有流程）。
- **v6 client** 调 v5 core：v6 client 调新路径 → 404（v5 core 不识别）；调老路径（/v1/chat）→ 正常工作。v6 client 必须配 v6 core / v6 gateway。
- **v6 core / v6 gateway** 自身：旧路径保留（GET /v1/agents / POST /v1/chat）；新增 9 个路径。

### 5.9 错误处理

| 阶段    | 错误                          | 处理                                              |
| ------- | ----------------------------- | ------------------------------------------------- |
| Build   | DB schema 损坏                | core 启动失败（pino log + 退出非 0）              |
| Startup | DB 文件不可写                 | core 启动失败（pino log + 退出非 0）              |
| Runtime | agent 名重复                  | client 端校验（POST /v1/agents 400 invalid_body） |
| Runtime | sessionId 不存在              | 404 agent_not_found                               |
| Runtime | agentId 不存在                | 404 agent_not_found                               |
| Runtime | llm_provider 非 mock（v6）    | 501 not_implemented                               |
| Runtime | LLM 调用异常                  | 500 internal_error + log 详细                     |
| Runtime | 鉴权失败（X-Client-Key 无效） | 401 unauthorized（v3 鉴权）                       |
| Runtime | gateway 不可达                | client 端走 v3 既有网络错误处理                   |
| Test    | DB 迁移测试                   | v6 起新增 SQLite 迁移测试（schema 升级兼容性）    |

**核心原则**（与 v1-v5 一致）：

- **build / startup loud fail**：schema 错、DB 不可写 → 显式退出。
- **runtime soft fail**：业务错误走 4xx / 5xx + 中文友好提示（v5 friendlyApiError 已支持）。
