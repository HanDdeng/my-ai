# v2 — 三端版本同步与兼容机制

> 状态：🟡 待用户审阅
> 日期：2026-06-06
> 分支：`dev-20260606`
> 上一版本：[v1 — 项目架构与工程化骨架](./v1.md)

## 一、问题陈述

v1 收尾时三端均无版本机制，无法检测、无法预防"客户端、网关、核心安装版本不一致"的场景。v2 上线后这一问题会真实出现：

- **典型场景 A**：用户先升级 client v2.0，gateway 仍跑 v1.5。client 调用了 v2 才有的 endpoint，得到 404 或字段缺失，行为静默错误。
- **典型场景 B**：运维升级 core v2.0，gateway 仍跑 v1.5（忘了同步重启）。core 改了 `/agents` 响应 schema，gateway 按旧 schema 透传给 client，client 解析失败。
- **典型场景 C**：三端升级到不同步的状态，用户问"现在跑的是哪一版？"，需逐台机器手动查。

v2 的任务：**让三端的版本号可被检测、范围可被声明、不匹配可被定位**。

## 二、目标 / 非目标

### 目标

- **可检测**：每端暴露自己的版本与上游兼容范围，握手可发现不匹配。
- **可声明**：顶层单一来源（`versions/compat-matrix.json`），build-time 同步到各子项目。
- **不阻断**：v2 阶段所有不匹配场景都走"软警告"路径，不因版本不一致导致应用完全不可用。
- **可演进**：v2 默认 gateway + core 同包同版本（一次 release 三端齐升），但 schema 结构允许 v3+ 拆分发版（n:n 矩阵），无需重写代码。

### 非目标

- **不做 feature intersection / capability negotiation**：版本范围匹配即认为可用，不在握手阶段计算"功能子集"（用户决策：只取范围，不计算 feature）。
- **不做 v1 兼容**：v1 没有 compat 字段被视为"未声明"，按"未声明 = 拒握手"处理（v1 自然淘汰，v2 重新计版本）。
- **不做自动升级依赖**：v1 收尾追加的 Dependabot 行为不同步问题（见 v1.md 第 5 项）保持现状，v2 不解决。
- **不做 Tauri 代码签名 / OpenAPI 文档 / 客户端组件测试空白修复**：这些都是 v1 待办，与版本机制正交，留给后续版本。

## 三、现状回顾

v1 收尾时各子项目均为 `version: "0.0.0"` 且 `private: true`，没有任何版本机制：

- 三个子项目 `package.json` 的 `version` 字段均为占位值；client 的 `tauri.conf.json` 也未设置 version
- `/health` 端点（client → gateway、gateway → core）只返回 `{ok, service}`，无版本信息
- CI 只产出三个独立安装包/服务，之间没有任何"组合校验"
- 用户安装形态：client 走 Tauri 安装包（GUI），gateway 与 core 各跑一个 Node 进程（CLI / 服务）

这种状态下，"三端版本不匹配"既不可检测也不可预防。

## 五、最终设计

### 5.1 架构总览

核心思想：**"连通性"和"版本"是两个独立阶段**，叠加在现有 HTTP 链路上。

```
client (Tauri)                              gateway                          core
     │                                          │                              │
     │  ①  PAIRING（连通性）                    │                              │
     │  GET /health (用户填的URL)               │                              │
     │ ────────────────────────────────────────► │                              │
     │                                          │  GET /health (liveness)      │
     │                                          │ ────────────────────────────►│
     │                                          │ ◄────────────────────────────│
     │ ◄────────────────────────────────────────│                              │
     │  { ok, service, version, schema }        │                              │
     │                                                                  ↓
     │  ②  HANDSHAKE（版本校验）                ↓
     │  本地 compat.upstream.gateway ⊇ version?
     │  ✓ in range → 无动作（HEALTHY）
     │  ✗ out of range → 状态置为 MISMATCH，banner 提示
     │
     │  ③  HEARTBEAT（5 min 间隔）
     │  GET /health → 重复 ②
```

**关键状态机（client 端）**：

```
INIT
  ↓ 用户填 URL + 触发 ping
PAIRING (transient) → 成功: PAIRED (立即进入 handshake), 失败: PAIR_FAILED
  ↓ 进入 PAIRED 状态，首次 handshake
HEALTHY   ←── version ⊇ compat range
MISMATCH  ←── version ⊄ compat range (banner 显示，可关)
   ↓ heartbeat 重检通过: HEALTHY
   ↓ heartbeat 重检失败: 仍 MISMATCH（banner 已被关时不再重亮）
PAIR_FAILED  ←── 网络/5xx/超时
```

### 5.2 顶层数据模型

`versions/compat-matrix.json` 是 source of truth。

```json
{
  "schema": 1,
  "components": {
    "client": { "version": "0.0.2" },
    "gateway": { "version": "0.0.2" },
    "core": { "version": "0.0.2" }
  },
  "compat": {
    "client": { "gateway": ">=0.0.2 <0.1.0" },
    "gateway": { "core": ">=0.0.2 <0.1.0" },
    "core": {}
  }
}
```

**字段语义**：

- `schema`：当前 schema 版本号；CI 校验写入时必须为 1，未来 schema 变更时递增
- `components.<name>.version`：该子项目当前 semver
- `compat.<downstream>.<upstream>`：downstream 接受的 upstream 版本范围（semver range 语法）

**v2 阶段"同包同版本"的体现**：所有 `components.*.version` 字段在 release 时由同一动作统一写入。运行时仍按"上游版本 ⊇ 下游范围"独立判断，与是否捆绑无关——为 v3+ 拆分分发留口子。

**部署形态（同一套结构支持 3 种）**：

| 形态                             | 三端关系                    | 握手                                                            | 何时启用                                       |
| -------------------------------- | --------------------------- | --------------------------------------------------------------- | ---------------------------------------------- |
| v2 捆绑（同 release）            | 三个 version 字段一致       | client↔gateway 握手，gateway↔core 握手**不实现**（schema 预留） | v2 默认                                        |
| v3+ 半拆（gateway 独立 release） | gateway 单独发版，core 跟随 | client↔gateway + gateway↔core 都握手                            | 把 `gateway.compat.core` 调宽到合理 range 即可 |
| v3+ 全拆（三端独立 release）     | 三套 release pipeline       | 两条握手链各做各的                                              | matrix range 调宽到合理 range                  |

### 5.3 Distribution 机制：build-time 同步

**所有子项目都用 build-time 同步**，不读运行时顶层文件路径。理由：

- Tauri 客户端必须把 compat 打进 bundle，运行时无法读顶层文件
- 对称性：三端用同一种机制，CI 校验简单
- 生成的 `.compat.generated.json` 小到可以 grep，调试时容易看

**`scripts/sync-compat.mjs`**（新文件）：

```js
// 读 versions/compat-matrix.json，按 components 切出每端需要的子集。
// - gateway / core: 写到子项目根 .compat.generated.json（运行时 fs.readFileSync 读）
// - client:          写到 client/src/compat.generated.ts（编译时 import）
// 由各子项目 predev / prebuild 钩子触发，dev 期间改了 matrix 也立即生效。
import { readFileSync, writeFileSync } from 'node:fs';
import { resolve } from 'node:path';

const matrix = JSON.parse(readFileSync('versions/compat-matrix.json', 'utf8'));
const { components, compat } = matrix;

if (matrix.schema !== 1) {
  console.error(`✖ 不支持的 compat-matrix schema: ${matrix.schema}（当前仅支持 1）`);
  process.exit(1);
}

for (const [name, info] of Object.entries(components)) {
  const slice = {
    version: info.version,
    upstream: compat[name] ?? {},
  };

  if (name === 'client') {
    // 客户端编译时注入：转成 TS 常量，省 runtime 读文件
    const ts = [
      '// 自动生成，请勿手改（predev / prebuild 钩子覆盖）',
      '// 来源：versions/compat-matrix.json',
      `export const COMPAT = ${JSON.stringify(slice, null, 2)} as const;`,
      '',
    ].join('\n');
    writeFileSync(resolve('client/src/compat.generated.ts'), ts);
    console.log(`✓ client → client/src/compat.generated.ts`);
  } else {
    const dst = resolve(name, '.compat.generated.json');
    writeFileSync(dst, JSON.stringify(slice, null, 2) + '\n');
    console.log(`✓ ${name} → ${dst}`);
  }
}
```

**各子项目 package.json 钩子**：

```jsonc
{
  "scripts": {
    "predev": "node ../scripts/sync-compat.mjs",
    "prebuild": "node ../scripts/sync-compat.mjs",
  },
}
```

**`.gitignore` 新增**：

```
.compat.generated.json
```

**Tauri 客户端 compat 数据来源**：

客户端不做运行时读取——`predev` / `prebuild` 钩子跑 sync 脚本生成 `client/src/compat.generated.ts`，Vite 编译时内联进 JS bundle。运行期直接 `import { COMPAT } from './compat.generated'`。**无需** 把 `.compat.generated.json` 列入 `tauri.conf.json` 的 `bundle.resources`，也**不引入** `@tauri-apps/plugin-fs`。

**`tauri.conf.json` 仅设置 `version` 字段**（与 `client/package.json` 保持一致，CI 校验同步）：

```jsonc
{
  "version": "0.0.2",
}
```

**运行时加载**：

- gateway / core 启动时 `fs.readFileSync` 读 `dist/../.compat.generated.json`
- Tauri 客户端编译时 import `src/compat.generated.ts`（Vite 内联）

### 5.4 端点形态：扩展 `/health`，不新增 `/version`

理由：

- 现有 `/health` 已被 client 启动时调用，扩字段零成本
- 老 client 忽略新字段，向后兼容（v1 客户端即使不升也能继续用旧字段）
- heartbeat 复用现有健康检查路径

**新的 `/health` 响应**：

```json
{
  "ok": true,
  "service": "gateway",
  "version": "0.0.2",
  "schema": 1
}
```

**字段语义**：

- `ok`：liveness 标志（5xx 即 false，不影响 version 字段返回）
- `service`：端点名（`"gateway"` / `"core"`）
- `version`：semver
- `schema`：compat-matrix schema 版本号

**liveness ≠ 版本兼容**：core 5xx 时 `ok: false` 但 `version` 仍正确返回；client 看到 `ok: false` 走"PAIR_FAILED"分支，看到 `ok: true` 但 version 不在范围内走"MISMATCH"分支。

### 5.5 Handshake + Heartbeat 协议

**Handshake 触发点**：

- 客户端 PAIRING 成功后立即执行一次
- 此后每 5 分钟（`VITE_HEARTBEAT_INTERVAL_MS`，默认 `300000`）触发一次

**Handshake 逻辑**：

```ts
// client/src/compat/handshake.ts（伪代码）
import { checkCompat } from '@my-ai/compat'; // 抽到 shared
import COMPAT from './.compat.generated.json';

async function handshake(gatewayUrl: string): Promise<Status> {
  let res: Response;
  try {
    res = await fetch(`${gatewayUrl}/health`);
  } catch {
    return 'PAIR_FAILED';
  }
  if (!res.ok) return 'PAIR_FAILED';
  const { version, schema } = await res.json();
  if (schema !== 1) return 'MISMATCH'; // 未知 schema 视作不兼容
  return checkCompat(version, COMPAT.upstream.gateway) ? 'HEALTHY' : 'MISMATCH';
}
```

**Version range checker 实现方式**：函数体仅 4 行（valid + validRange + satisfies + includePrerelease），三端各持一份**等价实现**，避免新建一个 monorepo 包带来的工作区配置、TS project reference、catalog 共享收益与成本不匹配。规则：

- canonical 版本在 `core/src/compat/check.ts`（canonical = 跑测试最多的一端）
- `gateway/src/compat/check.ts` 和 `client/src/compat/check.ts` 各复制一份，文件首行注释 `// 与 core/src/compat/check.ts 保持一致`
- 三端各自的单元测试覆盖完全相同的用例集（一份 fixture 复制三处）
- 后续若函数体增长（> 10 行）、或加新端（desktop / mobile），才考虑抽到 `packages/compat/`

依赖：`semver` 加入根 `pnpm-workspace.yaml` 的 `catalog` 块（目前未列入），三端通过 `catalog:` 引用。

```ts
import semver from 'semver';

export function checkCompat(got: string, want: string): boolean {
  if (!semver.valid(got) || !semver.validRange(want)) return false;
  return semver.satisfies(got, want, { includePrerelease: true });
}
```

**Heartbeat 行为**：

- 在 `useEffect` 里 `setInterval(handshake, VITE_HEARTBEAT_INTERVAL_MS)`
- 失败时不把 HEALTHY 变 PAIR_FAILED（避免网络抖动导致状态跳变）
- 失败仅记日志，下个 interval 重试

### 5.6 客户端状态机（React）

```tsx
// client/src/App.tsx（核心结构）
type Status = 'PAIRING' | 'HEALTHY' | 'MISMATCH' | 'PAIR_FAILED';

function App() {
  const [status, setStatus] = useState<Status>('PAIRING');
  const [bannerDismissed, setBannerDismissed] = useState(false);
  const [lastVersion, setLastVersion] = useState<string | null>(null);

  // 启动时 + 每 5 min 触发一次 handshake
  useEffect(() => {
    let cancelled = false;
    const run = async () => {
      const next = await handshake(gatewayUrl);
      if (cancelled) return;
      setStatus(next);
      if (next === 'HEALTHY') setLastVersion(await fetchVersion(gatewayUrl));
    };
    void run();
    const id = setInterval(run, VITE_HEARTBEAT_INTERVAL_MS);
    return () => {
      cancelled = true;
      clearInterval(id);
    };
  }, [gatewayUrl]);

  return (
    <>
      <MainUI />
      {status === 'MISMATCH' && !bannerDismissed && (
        <MismatchBanner
          gatewayVersion={lastVersion}
          requiredRange={COMPAT.upstream.gateway}
          onDismiss={() => setBannerDismissed(true)}
        />
      )}
    </>
  );
}
```

### 5.7 Banner 行为规则

| 场景                                            | banner 行为                                                       |
| ----------------------------------------------- | ----------------------------------------------------------------- |
| 启动直接 mismatch                               | 显示（bannerDismissed 初始 false）                                |
| 启动 mismatch → 关闭 → 持续 mismatch            | 保持收起                                                          |
| 启动 mismatch → 关闭 → 变 healthy → 又 mismatch | **保持收起**（dismissed 是 session 级 sticky）                    |
| 进程重启                                        | 重置为 false，下一次 mismatch 显示                                |
| 关闭按钮被点                                    | dismissed = true，**不再被任何状态变化**重新点亮（同 session 内） |

**关键**：`bannerDismissed` 是 React useState 局部状态，**不写本地存储**。进程退出状态丢失。

**Banner 文案**：

```
⚠️ Gateway v1.5.0 超出 client 兼容范围 (>=0.0.2 <0.1.0)。
   部分功能可能不可用，建议升级 gateway。
   [查看升级指南]  [关闭]
```

`test` 函数行为（与 `useEffect` 内的 handshake 逻辑共用）：

```ts
const test = async () => {
  setStatus('PAIRING');     // 显式回到 transient
  const next = await handshake(gatewayUrl);
  setStatus(next);
  // 注意：点击 test 不重置 bannerDismissed。
  // 用户已关闭 banner 的本意是"我已知道 mismatch，别再打扰"，retry 不应重亮。
};
````

**为何 retry 不重置 bannerDismissed**：用户点关闭是"我已知晓 mismatch"的明确意图；点"测试"重试是"我想知道现在状态如何"，两者语义独立。retry 期间状态显示会从 PAIR_FAILED → PAIRING → HEALTHY/MISMATCH 流转，但 banner 仍按 session sticky 规则。

````

### 5.9 发布流程：per-subproject version bump + GitHub Release

**核心规则**：

> 改了某子项目代码 → 该子项目 version 必须变；不变 → CI 拒合入

具体：
- 改了 `client/src/**` 或 `client/src-tauri/**`（不含 version 文件本身） → `client/package.json` 与 `client/src-tauri/tauri.conf.json` 两个 version **必须同步修改**
- 改了 `gateway/src/**` → `gateway/package.json` version 必须修改
- 改了 `core/src/**` → `core/package.json` version 必须修改
- 只改 `versions/compat-matrix.json` → **不要求** version bump（policy 文件）
- 只改 `versions/*.md`、`.github/**`、根目录工具链配置（prettier / eslint / husky）→ **不要求** version bump

**Release 触发**：每次 PR 合入 main（`--squash` 合并，参照 `feedback_merge_rule.md`）后，CI 跑 release job：
1. 比对上次 release 与当前 main 的 version 文件
2. 任意子项目 version 变了 → 创建 GitHub Release
3. 仅 `versions/compat-matrix.json` 变化（无 version bump）→ 仍出 release（policy 变化算有效发布）
4. 完全无变化（仅文档/CI/根工具链）→ 不出 release

**Release tag 格式**：`release-YYYYMMDD-{short-sha}`（如 `release-20260606-a1b2c3d`）

- gateway.compat.core: "=0.0.2" → ">=0.0.2 <0.1.0"
````

**派生规则**：

- 顶部子项目版本、兼容矩阵从 `versions/compat-matrix.json` 读
- changelog 条目从该次 release commit 的 `git diff` 文件路径推断子项目归属
  - `client/**` → client
  - `gateway/**` → gateway
  - `core/**` → core
  - 其他（`.github/**`、根工具链、`versions/*.md`）→ 不计入 changelog
- 按 commit type 前缀分桶：`feat:` → 功能变更，`fix:` → Bug 修复，其他（`chore:` / `docs:` / `ci:` / `refactor:` 等）→ 静默
- 渲染先按 type 一级、再按子项目二级
- 某子项目在某分类下无变更 → 显式显示"（无）"
- 跨子项目 commit（diff 同时涉及 client 与 gateway）→ 分桶到两个子项目

**commit subject 中英混排约定**（追加到 commitlint 文档）：

- boilerplate（"子项目版本"、"功能变更"等）中文
- 技术名词、子项目名、commit type 前缀、commit hash、semver range、JSON 字段路径、release tag → 英文
- commit subject 主体由作者决定，建议中文以便 release notes 一致
- 不在 CI 自动翻译 commit subject

### 5.11 错误处理：build / startup / runtime 三档

| 阶段           | 错误                                                                         | 处理                                                                    |
| -------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Build / CI** | `versions/compat-matrix.json` 不是合法 JSON                                  | CI 红（`compat-matrix-check` job）                                      |
| **Build / CI** | `compat-matrix.json` 缺 `schema: 1` 字段                                     | CI 红                                                                   |
| **Build / CI** | `compat[name]` 里的 name 不在 `components` 里                                | CI 红                                                                   |
| **Build / CI** | `compat.*` 里的 range 不是合法 semver range                                  | CI 红                                                                   |
| **Build / CI** | PR 改了子项目代码但未 bump 对应 `package.json`                               | CI 红（`version-bump-check`）                                           |
| **Build / CI** | client 改了代码但 `tauri.conf.json` version 与 `package.json` version 不一致 | CI 红                                                                   |
| **Startup**    | 子项目找不到 `.compat.generated.json`                                        | 进程**直接退出 1**，日志 "missing .compat.generated.json; run prebuild" |
| **Startup**    | `.compat.generated.json` 不是合法 JSON                                       | 进程**直接退出 1**                                                      |
| **Startup**    | `.compat.generated.json` 里 `version` 字段不是合法 semver                    | 进程**直接退出 1**                                                      |
| **Runtime**    | client 调 gateway `/health` 失败（网络 / 5xx / 超时）                        | 状态 → `PAIR_FAILED`，UI 红点，banner 不显示                            |
| **Runtime**    | gateway 收到 core 的 `/health` 但 `version` 字段缺失                         | log warn，handshake 视作"未声明"，按 mismatch 处理                      |
| **Runtime**    | 客户端解析 `/health` 的 `version` 失败                                       | 状态 → `MISMATCH`（保守路径：unparseable = unknown = 不在范围内）       |
| **Runtime**    | heartbeat 调 `/health` 时网络中断                                            | 不改 status，下个 heartbeat 重试（避免 HEALTHY ↔ PAIR_FAILED 抖动）     |

**核心原则**：build / startup 阶段**loud fail**（早暴露问题），runtime 阶段**soft fail**（不中断用户）。

### 5.12 CI 校验 job 清单

#### Job A：`compat-matrix-check`（新增，独立 workflow）

```
name: compat-matrix schema check
runs-on: ubuntu-latest
steps:
  - checkout
  - setup pnpm + node
  - install
  - run: node scripts/check-compat-matrix.mjs
```

`scripts/check-compat-matrix.mjs` 做 5 项校验（任一不通过即非零退出）：

1. `JSON.parse(matrixFile)` 成功
2. `matrix.schema === 1`
3. `matrix.components` 至少含 `client` / `gateway` / `core`
4. 对每个 `compat[downstream][upstream]`，`semver.validRange(range)` 返回非 null
5. `compat[downstream]` 引用的 upstream 必须在 `components` 里存在

#### Job B：`version-bump-check`（新增，独立 workflow）

在 `quality` job 之后、`build-*` job 之前跑：

```bash
changed=$(git diff --name-only origin/main...HEAD)

# client
client_code=$(echo "$changed" | grep -E '^client/(src|src-tauri/.*\.(rs|toml))$' || true)
client_pkg=$(echo "$changed" | grep -E '^client/package\.json$' || true)
client_tauri=$(echo "$changed" | grep -E '^client/src-tauri/tauri\.conf\.json$' || true)
if [ -n "$client_code" ] && { [ -z "$client_pkg" ] || [ -z "$client_tauri" ]; }; then
  fail "client 代码有变更但未同步 bump client/package.json 和 client/src-tauri/tauri.conf.json"
fi

# gateway / core（仅需 bump package.json）
for sub in gateway core; do
  code=$(echo "$changed" | grep -E "^$sub/src/" || true)
  pkg=$(echo "$changed" | grep -E "^$sub/package\.json$" || true)
  if [ -n "$code" ] && [ -z "$pkg" ]; then
    fail "$sub 代码有变更但未 bump $sub/package.json"
  fi
done
```

**豁免清单**（不要求 version bump）：

- `versions/compat-matrix.json`（policy 文件）
- `versions/*.md`（版本说明文档）
- `.github/**`（CI 配置）
- 根目录 `package.json` 工具链升级（prettier / eslint / husky）
- 根目录 `.editorconfig` / `.prettierrc.json` / `eslint.config.mjs` 等
- 仓库级 README / LICENSE

#### Job C：`release`（新增，独立 workflow）

```
name: GitHub Release
runs-on: ubuntu-latest
if: github.event_name == 'push' && github.ref == 'refs/heads/main'
needs: quality
steps:
  - checkout (fetch-depth: 0 + fetch-tags)
  - 读上次 release tag：last=$(git describe --tags --abbrev=0 2>/dev/null || echo "")
  - 解析本次 diff
  - 判断是否需要 release
  - 调用 softprops/action-gh-release@v2 创建 release
  - body 模板从 versions/compat-matrix.json 渲染
```

判断逻辑：

- 若 `last` 为空（**首次 release**）→ 强制出 baseline release：tag + body，但 body 中"功能变更 / Bug 修复 / 兼容性变更"段全部"（无）"（无 diff 来源），仅含 matrix 快照与子项目版本
- 否则 diff 中含 `versions/compat-matrix.json` 改动 → 出 release（policy 变化）
- 否则 diff 中四个 version 文件任一有改动 → 出 release
- 否则 → 不出 release

#### 修订：现有 `quality` job 顺序

```
quality
  ├─ typecheck
  ├─ lint
  ├─ test
  ├─ dedupe check
  ├─ secret scan
  └─ compat-matrix-check（新增至此）
```

`version-bump-check` 不放 quality 里，**作为独立 job** 跑在 quality 之后、build-\* 之前。参考 v1 dependabot 经验：独立 job 容易定位失败原因。

### 5.13 测试策略

#### 单元测试

**(a) `scripts/sync-compat.mjs`**（`scripts/sync-compat.test.mjs`，vitest node 环境）

用例：

- 给定完整 matrix，生成 3 个子项目 slice，文件内容精确匹配期望
- 缺 `compat[downstream]` 字段 → 该子项目 `upstream: {}`
- 顶层 `schema` 不是 1 → 脚本拒绝执行
- 写出的 JSON 末尾有换行（prettier 一致性）
- 子项目目录下已有旧 `.compat.generated.json` → 覆盖写入

**(b) `scripts/check-compat-matrix.mjs`**（`scripts/check-compat-matrix.test.mjs`）

用例：

- 缺 `schema` 字段 → fail
- `schema` 不为 1 → fail
- 引用不存在的 component → fail
- 非法 range 字符串 → fail
- 正常 matrix → pass

**(c) version range checker**（`core/src/compat/check.ts` + `check.test.ts`）

用例：

- `checkCompat("0.0.2", ">=0.0.2 <0.1.0")` → true
- `checkCompat("1.5.0", ">=0.0.2 <0.1.0")` → false
- `checkCompat("3.0.0", ">=0.0.2 <0.1.0")` → false（边界）
- `checkCompat("0.0.2-rc.1", ">=0.0.2 <0.1.0")` → true（pre-release 在范围内）
- `checkCompat("not-a-version", ">=0.0.2")` → false（保守 false）
- `checkCompat("0.0.2", "not-a-range")` → false
- `checkCompat("0.0.2", "")` → false
- `checkCompat("", ">=0.0.2")` → false

**(d) `/health` 端点**（`core/src/routes/health.test.ts` 已有，扩展；`gateway/src/routes/health.test.ts` 新增）

扩展用例：

- 响应含 `version` 字段，值与 `package.json` version 一致
- 响应含 `schema: 1`
- 5xx 时 `ok: false` 但 `version` 字段仍正确（liveness 与版本独立）

#### 集成测试（vitest + fastify.inject）

**(e) gateway 启动时 handshake**（v2 阶段跳过此用例）

v2 阶段不实现 gateway↔core handshake，仅在 v3+ 拆分时启用。用例留作占位：

- mock 一个返回 version="1.0.0" 的 core，gateway 启动时 `compat.core = ">=0.0.2 <0.1.0"` → 启动失败
- mock 返回 version="0.0.2" → 启动成功

#### 客户端组件测试（vitest + jsdom + Testing Library）

**(f) 状态机**（`client/src/components/Settings.test.tsx` 或 `App.test.tsx`）

用例：

- 初始 render → `status=PAIRING`
- mock 成功 `/health` 返回 version 在范围内 → `status=HEALTHY`，banner 不可见
- mock 成功 `/health` 返回 version 在范围外 → `status=MISMATCH`，banner 可见
- banner 出现后点关闭按钮 → banner 消失
- banner 关闭后 mock 再次 mismatch → banner 仍不可见（session sticky）
- mock 网络失败 → `status=PAIR_FAILED`，banner 不可见
- 渲染中 status 文本正确（"配对成功 · gateway v0.0.2" 等）

**(g) heartbeat**（vitest `vi.useFakeTimers`）

用例：

- 启动后 fake timer 推进 5 min，handshake 被再次调用
- heartbeat 失败时 status 不变（不抖到 PAIR_FAILED）
- heartbeat 成功且 status 从 HEALTHY 变 MISMATCH 时，bannerDismissed 仍为 true 时 banner 不重亮

#### 手工 E2E（不进 CI，每次 release 前冒烟）

- 装好三端，启动 client，配 gateway URL
- 验证 HEALTHY 状态显示
- 手动改 gateway package.json version 模拟"用户升级了 gateway 但没通知 client"，不重启 gateway
- 观察 client heartbeat 后是否检测到 mismatch + banner 显示
- 关闭 banner，等待下一次 heartbeat，确认 banner 不重亮
- 重启 client，验证 banner 重新显示
- 故意把 `versions/compat-matrix.json` 写坏，CI 跑 `compat-matrix-check` 看是否红
- 故意 PR 不 bump version，CI 跑 `version-bump-check` 看是否红

## 七、回滚与兼容

**回滚 v2 自身**：若 v2 整体不可用，回滚 commit 即可。所有 v2 引入的文件随 revert 一起消失；现有 v1 工程化骨架（构建、CI、ESLint、Prettier）不受影响。

**对 v1 老安装的兼容**：v1 客户端、gateway、core 均无 compat 字段。v2 升级后 handshake 视"无 compat 声明"为 mismatch，banner 会显示并要求升级。**这意味着 v1 任何一端如果仍在用户侧运行，与 v2 任何一端的握手都会失败**。v2 团队需在 release notes 明确提示"v1 升级 v2 必须三端同步"。

**v2 → v3 兼容性**：v2 release 写入的 compat range `>=0.0.2 <0.1.0` 在 v3 release 时可保留（向下兼容），也可调宽到 `>=0.0.2 <0.2.0`（v3 gateway 接受 v2 core）。matrix schema 若有 breaking 变更需递增 `schema` 字段。
