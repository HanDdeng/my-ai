## 一、问题陈述

v5 合入后（commit `29971b7`）core 端（v1-v5 现状）：

- **单一硬编码 `EchoAgent`** 在 `buildServer()` 构造期注册（v1 设计）；无 CRUD、无持久化、用户不可配。
- **`createLLMClient` 用 `switch (cfg.LLM_PROVIDER)`**，全局单 provider；`Config.LLM_PROVIDER` / `LLM_BASE_URL` / `LLM_MODEL` 走 env。
- **session 概念是"客户端透传字段"**——`/v1/chat` 接 `sessionId` 但 core 不存；core 端**无 DB**。
- **`MockLLMClient` 是唯一 LLM 实现**——不接任何真实 LLM 服务。
- README "后续计划"首项 "core 接入 LLM（OpenAI 兼容协议）"——v1 仅 mock，未真接。

v6.md（已合入 `5ab702d`）的 core 部分定调了 7+ 块改造：

- DB 三表（`agents` / `sessions` / `messages`）+ DAO + 迁移
- 11 个 API 端点（10 个 v6 新增 + 1 个 GET `/v1/agents` 改造）
- `createLLMClient` 拆 `Map<provider, factory>` 可插拔
- `AgentRegistry` 改空 + lazy 实例化
- 删 `EchoAgent` 硬编码
- gateway→core `X-Internal-Client-Key` 内部 header
- 5.9 错误码体系

v6.md 是**三端全景**设计文档，core 部分散落在 5.1-5.10 + 第六章 + 第八章。**v6.1 的任务**：

- 把 v6 core 部分从 v6.md 抽出为独立的设计文档
- 对 v6 文字未锁的粒度（API body schema / 错误码细化 / DB 索引 / 类与 env 变量的去留 / 字段集 / 协议语义）做微调并**锁到实施级**
- gateway 与 client 部分**不**展开，留待后续 v6.2+ 子版本

**v6.1 仅是设计文档**——本版本**不**含实现计划、不**规划** v6.1 实施顺序。v6.1 范围里的具体代码实施按"DB / 路由 / 工厂 / 客户端组件"维度拆给后续各自承接。

## 二、目标 / 非目标

### 目标

- **core 端职责清晰**：agent / session / message 持久化的**权威源**（新增 SQLite 四张表，含 `schema_version`）；registry 完全删除；agent 实例化按"收到 chat 时 lazy 构造"走 DB；新增 10 个 CRUD 端点 + 同步 chat 路由（POST `/v1/sessions/{id}/messages` 取代 v1 `/v1/chat`）。
- **完整 CRUD**：9 个 agent 字段（`name` / `description` / `llm_provider` / `base_url` / `model` / `max_tokens` / `enabled_api` / `system_prompt` / `capabilities`）；DB `name` UNIQUE 约束；POST 重复走 409 `agent_name_conflict`。
- **session 持久化**：core 端按 session 持久化 messages 表；`agent_id` 字段**不可改**（切 agent = 开新 session）；`client_key` 字段保留作"创建者"审计（**不**作权限——一人多设备场景下不同 clientKey 可访问同一 session）。
- **真实接入 LLM**：v6.1 实现 `OpenAICompatibleLLMClient`（走 OpenAI 协议，可对接任何兼容 OpenAI chat completions 接口的服务——本地 Ollama / vLLM / LM Studio / 第三方代理等，**不限定**为 OpenAI 公司服务）；`createLLMClient` 重构为 `Map<provider, factory>`。
- **`MockLLMClient` 仅供 CI 保留**：不出现在 `agents.llm_provider` CHECK 约束（`CHECK IN ('openai-compatible')`）；测试通过依赖注入直接构造 `MockLLMClient`，不走 factory。
- **API key 走 env，可选**：core 启动读 `LLM_API_KEY` env；不设则不发 `Authorization` header（适用于本地无鉴权服务）；agent 行**不存**任何 key 字段。
- **`enabled_api` 预留字段**：当前 v6.1 阶段 core **不消费**此字段；后续 评估"为其他系统提供 API 调用指定 agent 直接返回结果"能力。
- **i18n 不涉及**：core 端 i18n 沿用现状（pino 日志英文、错误码英文）；client 端在 `friendlyApiError` 内做码→本地化文案映射（v5 现状）。
- **v5 兼容**：v5 client / v5 gateway / v5 core 的现有路径与流程全部保留；v6.1 增的 API 对 v5 透明。
- **零网络协议破坏**：v6.1 改的是"加新端点" + "core 内部数据流"；`X-Client-Key` 鉴权、响应 `{ok, data, code, message}` 包装、版本握手 compat 全部不变。

### 非目标

- **不**实现其他 LLM provider：`anthropic` / `ollama` / 自研 provider **不**在 v6.1 范围；`agents.llm_provider` CHECK 仅 `openai-compatible`；后续 加新 provider 走 schema 迁移。
- **不**做流式 chat 响应：v6.1 仍为同步（沿用 v1 chat 路由语义）。
- **不**做 agent 软删除 / archive / 启用-停用开关：v6.1 删除即真删（CASCADE）；`enabled_api` 不作 agent 启停语义。
- **不**做 agent 标签 / 分组 / 搜索：v6.1 不做。
- **不**做权限模型：session 跨 clientKey 共享；任何已配对的 clientKey 都能 GET/POST 同一 session 的 messages；`sessions.client_key` 仅作创建者审计字段。
- **不**做多 agent 弹窗并存：客户端 UI 与 v6.1 正交，留给后续 v6.2+ 评估。
- **不**做 plan 写 / 实现顺序：v6.1 仅出设计文档；v6.1 范围里具体代码实施按维度拆给后续。
- **不**改 compat 矩阵范围：v6.1 涉及 core 单端代码改动，**预留 v6.1 实际写代码时由实施 plan 按 v2 规则 bump core 版本**（v6.1 文档不锁 bump 数字）。
- **不**改 v1-v5 任何已有功能的行为：v1 chat 路由保留、v2 compat 保留、v3 鉴权保留、v4 测试目录保留、v5 i18n 保留（i18n 在 client 端，不在 core）。
- **不**改 v6 中 gateway / client 部分的设计：v6.1 仅细化 core；gateway 透传与 client UI 仍按 v6.md 5.1 / 5.5 / 第六章描述，不在 v6.1 重写。
- **不**实现 client 端"虚拟办公室" UI：v6.1 是后端设计；client 端 UI 改造由后续 v6.2+ 承接（client 可调 v6.1 新增的 10 个 API）。

**v5 → v6.1 代码层面变化**（core 端）：

```
v5: buildServer() 构造期 registry.register(new EchoAgent(llm))
v6.1: buildServer() 构造期
       1. 打开 SQLite (./core.db)
       2. 跑 schema_version 检查 (期望 version=1，缺失则启动失败)
       3. 启动 Fastify（registry 类已删除；agent 按需 lazy 实例化）
```

**v5 → v6.1 数据流变化**（core 内部）：

```
v5  POST /v1/chat:
   core 收 {agentId, sessionId, content}
   → registry.get(agentId) → EchoAgent
   → EchoAgent.run(...) → 同步返回 reply
   → 不持久化

v6.1  POST /v1/sessions/{sid}/messages:
   core 收 {id, content} (含 X-Internal-Client-Key)
   → DB.sessions.get(sid) → 拿 agent_id + client_key (创建者审计)
   → DB.agents.get(agent_id) → 拿 9 字段
   → DB.messages.listBySession(sid) 按 id 字典序 → 拼上下文
   → OpenAICompatibleLLMClient(base_url, model, max_tokens, system_prompt, env.LLM_API_KEY)
        .chat({messages: [system?, ...history, new]})
   → DB insert user message (id 来自 client) + assistant message (id 由 core 生成)
   → 同步返回 {userMessage, assistantMessage}
```

### 5.2 数据模型（core 端 SQLite）

```sql
-- agents 表：用户 CRUD 的对象
CREATE TABLE agents (
  id            TEXT    PRIMARY KEY,
  name          TEXT    NOT NULL UNIQUE,                 -- client 生成 UUID v4；name 唯一
  description   TEXT    NOT NULL DEFAULT '',
  llm_provider  TEXT    NOT NULL DEFAULT 'openai-compatible',
  base_url      TEXT    NOT NULL,                        -- OpenAI 兼容协议 base URL（含版本路径，如 'http://localhost:11434/v1'）；不限定为 OpenAI 公司服务
  model         TEXT    NOT NULL,                        -- 模型名（如 gpt-4o-mini / qwen2.5:7b）
  max_tokens    INTEGER,                                 -- 1 ≤ x ≤ 32000；NULL = 不限
  enabled_api   INTEGER NOT NULL DEFAULT 0,              -- BOOLEAN (0/1)；v6.1 core 不消费
  system_prompt TEXT    NOT NULL DEFAULT '',
  capabilities  TEXT    NOT NULL DEFAULT '[]',           -- JSON 数组（灵活；元素任意字符串）
  created_at    TEXT    NOT NULL,                        -- ISO8601
  updated_at    TEXT    NOT NULL,
  CHECK (llm_provider = 'openai-compatible'),
  CHECK (max_tokens IS NULL OR (max_tokens >= 1 AND max_tokens <= 32000)),
  CHECK (length(name) > 0 AND length(name) <= 64),
  CHECK (length(description) <= 256),
  CHECK (length(model) > 0 AND length(model) <= 128),
  CHECK (length(base_url) > 0 AND length(base_url) <= 512),
  CHECK (length(system_prompt) <= 8192)
);
CREATE UNIQUE INDEX idx_agents_name ON agents(name);
```

```sql
-- sessions 表：每个 session 绑一个 agent
CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,                          -- client 生成 UUID v4
  agent_id    TEXT NOT NULL,                             -- 不可改；切 agent = 开新 session
  client_key  TEXT NOT NULL,                             -- 创建者 clientKey（审计字段；不作权限）
  title       TEXT NOT NULL DEFAULT '',                  -- v6.1 留空；后续 评估"取首条消息前 N 字"
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL,
  FOREIGN KEY (agent_id) REFERENCES agents(id) ON DELETE CASCADE,
  CHECK (length(id) > 0),
  CHECK (length(agent_id) > 0),
  CHECK (length(client_key) > 0)
);
CREATE INDEX idx_sessions_agent_id    ON sessions(agent_id);
CREATE INDEX idx_sessions_client_key  ON sessions(client_key);
```

```sql
-- messages 表：会话上下文窗口
CREATE TABLE messages (
  id          TEXT PRIMARY KEY,                          -- client 生成 UUID v4（保证顺序）
  session_id  TEXT NOT NULL,
  role        TEXT NOT NULL,                             -- 'user' | 'assistant' | 'system'
  content     TEXT NOT NULL,
  created_at  TEXT NOT NULL,
  FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE,
  CHECK (role IN ('user', 'assistant', 'system')),
  CHECK (length(content) > 0)
);
CREATE INDEX idx_messages_session_id ON messages(session_id);
```

```sql
-- schema_version 表：DB 迁移追踪
CREATE TABLE schema_version (
  version     INTEGER PRIMARY KEY,                       -- v6.1 起步 = 1
  applied_at  TEXT NOT NULL                              -- ISO8601
);
```

**启动时 schema 处理流程**：

```
启动 core：
  1. 打开 SQLite (./core.db)
  2. 查 SELECT name FROM sqlite_master WHERE type='table'
     ├─ 包含 'schema_version' → 查 SELECT version FROM schema_version
     │     ├─ version = 1 → OK；跳过建表
     │     └─ version ≠ 1 → pino log "schema version mismatch: expected 1, got {N}" + process.exit(1)
     └─ 不包含 'schema_version' → 视为首次启动：
           1. 执行 schema.sql（创建 4 张表 + 索引）
           2. INSERT INTO schema_version (version, applied_at) VALUES (1, '<ISO8601>')
           3. 继续启动
```

**关键约定**：

- v6.1 阶段**不实现**自动迁移；任何 schema 变更（v6.2+）需要：手动删 `core.db` + 重启（**不**迁移 v6.1 数据）
- 启动时**只检查** `schema_version` 表存在 + `version=1`；**不**检查其他表的 schema 演进（SQLite 不支持 schema diff 检查）
- 异常 DB（表存在但 schema_version 缺失）→ 启动失败（提示"DB 损坏，请删除 ./core.db 后重启"）

**字段约束汇总**：

| 字段                   | 约束                            | 失败行为                                                               |
| ---------------------- | ------------------------------- | ---------------------------------------------------------------------- |
| `agents.name`          | UNIQUE + 1..64 chars + NOT NULL | UNIQUE 冲突 → 409 `agent_name_conflict`；长度不符 → 400 `invalid_body` |
| `agents.description`   | 0..256 chars                    | 400 `invalid_body`                                                     |
| `agents.llm_provider`  | 必须 = 'openai-compatible'      | 400 `invalid_body`                                                     |
| `agents.base_url`      | 1..512 chars                    | 400 `invalid_body`                                                     |
| `agents.model`         | 1..128 chars                    | 400 `invalid_body`                                                     |
| `agents.max_tokens`    | NULL 或 1..32000                | 400 `invalid_body`                                                     |
| `agents.enabled_api`   | 0 或 1                          | 400 `invalid_body`                                                     |
| `agents.system_prompt` | 0..8192 chars                   | 400 `invalid_body`                                                     |
| `agents.capabilities`  | 合法 JSON 数组（zod 校验）      | 400 `invalid_body`                                                     |
| `sessions.title`       | 0..256 chars                    | 400 `invalid_body`                                                     |

**约束**：

- **agent `id`**：client 生成 UUID v4；删除 agent → CASCADE 删 sessions / messages。
- **session `id`**：client 生成 UUID v4；`agent_id` 字段**不可改**（PATCH 不允许改 `agent_id`）。
- **session `client_key`**：明文存；v6.1 不做脱敏（v3 鉴权已上，DB 仅本机 core 访问）。
- **message 顺序**：用 `id`（UUID v4 字典序）保证，**不**依赖 `created_at`。
- **跨 clientKey 共享**：session 任何字段包括 `client_key` 都不参与权限；任何已配对 clientKey 都能 GET/POST 同一 session 的 messages。
- **真删**：v6.1 不做软删除 / archive / 启用-停用开关；`enabled_api` 不作 agent 启停语义。
- **id 不复用**：删 agent 不复用 id（客户端不感知）。

#### 5.3.3 错误码（v6.1 细化）

| code | message               | 触发                                                                      |
| ---- | --------------------- | ------------------------------------------------------------------------- |
| 0    | `ok`                  | 正常                                                                      |
| 400  | `invalid_body`        | zod 校验失败 / 字段长度超限 / 字段缺失                                    |
| 401  | `unauthorized`        | `X-Internal-Client-Key` 缺失（理论上 gateway 必传，core 拒绝 = 调试信号） |
| 404  | `agent_not_found`     | `agentId` 不存在                                                          |
| 404  | `session_not_found`   | `sessionId` 不存在                                                        |
| 409  | `agent_name_conflict` | `agents.name` UNIQUE 冲突                                                 |
| 500  | `internal_error`      | 兜底                                                                      |
| 501  | `not_implemented`     | 预留（v6.1 实际不触发；后续加新 provider 时未实现走此码）                 |
| 502  | `upstream_error`      | OpenAI 兼容服务不可达 / HTTP 5xx / 响应 schema 异常                       |

**与 v6.md 差异**：

- 拆 `agent_not_found` / `session_not_found`（v6.md 合为一个 404）
- 新增 `agent_name_conflict` 409（v6.md 走 400 `invalid_body`）
- `not_implemented` 501 在 v6.1 不触发（仅 `openai-compatible` CHECK）；保留码值供后续 schema 迁移后未实现 provider 用
- `unauthorized` 401 仅由 gateway 注入的 `X-Internal-Client-Key` 缺失触发；正常 client→gateway→core 链路下不应出现

### 5.4 核心数据流：发一条消息

```
client (虚拟办公室，agent 弹窗)
  │
  │ POST /v1/sessions/{sid}/messages
  │ header: X-Client-Key: <clientKey>  (v3 鉴权)
  │ body: { id: <umId>, content: "..." }
  │
  ▼
gateway (v3 鉴权 + 透传)
  │
  │ 内部 HTTP POST /v1/sessions/{sid}/messages
  │ header: X-Internal-Client-Key: <clientKey>  (v6.1 新增)
  │ body: { id, content }
  │
  ▼
core:
  1. 校验 X-Internal-Client-Key 非空（缺失 → 401 unauthorized）
  2. 校验 body (zod) → { id, content }
  3. DB.sessions.get(sid)
     ├─ 不存在 → 404 session_not_found
     └─ 存在 → 拿 agent_id
  4. DB.agents.get(agent_id)
     ├─ 不存在（理论 CASCADE 后不可能）→ 404 agent_not_found
     └─ 存在 → 拿 9 字段
  5. DB.messages.listBySession(sid) 按 id 字典序升序 → history
  6. 构造 LLM 客户端：
     llm = createLLMClient(agent.llmProvider, {
       baseUrl: agent.baseUrl,
       apiKey: cfg.LLM_API_KEY,    // 可选；undefined 不发 Authorization（cfg.LLM_API_KEY 来自 env LLM_API_KEY）
       model: agent.model,
       maxTokens: agent.maxTokens,
     })
  7. 拼 messages：
     msgs = []
     if (agent.systemPrompt) msgs.push({ role: 'system', content: agent.systemPrompt })
     msgs.push(...history, { role: 'user', content })
  8. llm.chat({ model: agent.model, messages: msgs, maxTokens: agent.maxTokens })
     ├─ 200 + 合法 OpenAI 响应 → 拿 reply.content
     ├─ HTTP 5xx / 网络错 → 502 upstream_error
     ├─ 响应 schema 异常 → 502 upstream_error
     └─ 其他 → 500 internal_error
  9. DB insert user message (id = umId 来自 client)
  10. DB insert assistant message (id = amId 由 core 生成 randomUUID())
  11. 同步返回 { userMessage, assistantMessage }
```

**`openai-compatible.ts`**（真实 HTTP 客户端）：

```ts
import type { ChatRequest, ChatResponse, LLMClient } from './types.js';
import { LLMUpstreamError } from './errors.js';

export type OpenAICompatibleConfig = {
  baseUrl: string; // e.g. 'https://api.openai.com' / 'http://localhost:11434/v1'
  apiKey?: string; // 可选；undefined → 不发 Authorization header
  model: string; // e.g. 'gpt-4o-mini' / 'qwen2.5:7b'
  maxTokens?: number; // e.g. 2048
};

export class OpenAICompatibleLLMClient implements LLMClient {
  constructor(private readonly cfg: OpenAICompatibleConfig) {}

  async chat(req: ChatRequest): Promise<ChatResponse> {
    const url = `${this.cfg.baseUrl.replace(/\/$/, '')}/chat/completions`;
    const headers: Record<string, string> = { 'Content-Type': 'application/json' };
    if (this.cfg.apiKey) {
      headers['Authorization'] = `Bearer ${this.cfg.apiKey}`;
    }
    const body = {
      model: req.model,
      messages: req.messages,
      max_tokens: req.maxTokens ?? this.cfg.maxTokens,
      stream: false,
    };

    let res: Response;
    try {
      res = await fetch(url, {
        method: 'POST',
        headers,
        body: JSON.stringify(body),
        signal: AbortSignal.timeout(60_000), // 60s 上限
      });
    } catch (e) {
      throw new LLMUpstreamError(`fetch failed: ${(e as Error).message}`);
    }

    if (!res.ok) {
      const text = await res.text().catch(() => '');
      throw new LLMUpstreamError(`HTTP ${res.status}: ${text.slice(0, 256)}`);
    }

    let json: unknown;
    try {
      json = await res.json();
    } catch {
      throw new LLMUpstreamError('response is not valid JSON');
    }

    // 解析 OpenAI chat completions 响应
    const content = (json as any)?.choices?.[0]?.message?.content;
    if (typeof content !== 'string') {
      throw new LLMUpstreamError('response missing choices[0].message.content');
    }
    const usage = (json as any)?.usage;
    return {
      content,
      usage:
        usage && typeof usage === 'object'
          ? {
              promptTokens: Number(usage.prompt_tokens) || 0,
              completionTokens: Number(usage.completion_tokens) || 0,
              totalTokens: Number(usage.total_tokens) || 0,
            }
          : undefined,
    };
  }
}
```

```ts
async function handlePostMessage(sid: string, umId: string, content: string) {
  const session = db.sessions.get(sid);
  if (!session) throw new HttpError(404, 'session_not_found');

  const agent = db.agents.get(session.agentId);
  if (!agent) throw new HttpError(404, 'agent_not_found'); // 理论 CASCADE 后不可能

  const history = db.messages.listBySession(sid); // 按 id 字典序

  // 构造 LLM 客户端（每次新实例；v6.1 不缓存）
  const llm = createLLMClient(agent.llmProvider, {
    baseUrl: agent.baseUrl,
    apiKey: cfg.LLM_API_KEY, // cfg.LLM_API_KEY 来自 env LLM_API_KEY；undefined → 不发 Authorization
    model: agent.model,
    maxTokens: agent.maxTokens ?? undefined,
  });

  // 拼 messages
  const messages: ChatMessage[] = [];
  if (agent.systemPrompt) messages.push({ role: 'system', content: agent.systemPrompt });
  messages.push(...history.map(m => ({ role: m.role, content: m.content })));
  messages.push({ role: 'user', content });

  // 调 LLM
  const reply = await llm.chat({
    model: agent.model,
    messages,
    maxTokens: agent.maxTokens ?? undefined,
  });

  // 写 DB
  const now = new Date().toISOString();
  db.messages.insert({ id: umId, sessionId: sid, role: 'user', content, createdAt: now });
  const amId = randomUUID();
  db.messages.insert({
    id: amId,
    sessionId: sid,
    role: 'assistant',
    content: reply.content,
    createdAt: now,
  });
  db.sessions.updateTimestamp(sid, now);

  return {
    userMessage: { id: umId, sessionId: sid, role: 'user', content, createdAt: now },
    assistantMessage: {
      id: amId,
      sessionId: sid,
      role: 'assistant',
      content: reply.content,
      createdAt: now,
    },
  };
}
```

```ts
app.setErrorHandler((err, _req, reply) => {
  if (err instanceof HttpError) {
    return reply.code(err.status).send({ data: null, code: err.status, message: err.code });
  }
  if (err instanceof LLMNotImplementedError) {
    return reply.code(501).send({ data: null, code: 501, message: 'not_implemented' });
  }
  if (err instanceof LLMUpstreamError) {
    app.log.error({ err }, 'LLM upstream error');
    return reply.code(502).send({ data: null, code: 502, message: 'upstream_error' });
  }
  // zod 校验失败（Fastify 自动抛 FastifyError.validation）
  if ((err as any).validation) {
    return reply.code(400).send({ data: null, code: 400, message: 'invalid_body' });
  }
  // 兜底
  app.log.error({ err }, 'unhandled error');
  reply.code(500).send({ data: null, code: 500, message: 'internal_error' });
});
```
