> 状态：🟡 待用户审阅
> 日期：2026-06-07
> 分支：`dev-20260607`

## 一、问题陈述

v2 合入后，client 端能跟 gateway 握手 + heartbeat，URL 输入 + 测试已经在 Settings 面板里落地，但**没有任何身份概念**——任何能访问 gateway 端口的 client 都能调全部业务 endpoint。

v3 之前的现实：

- gateway 仅做路由透传 + 版本校验，**完全无鉴权**
- 网关侧**无任何持久层**（client 端 URL 也未持久化）
- client 与 gateway 是"网络通就行"的关系，**无"我是不是被这台 gateway 信任的客户端"概念**

这套机制在单用户单机器场景下够用，一旦 gateway 暴露在网络（局域网 / 公网 / 反代）就完全失效：

- 任何能 ping 通 gateway 端口的机器都能调 `/v1/agents`
- 装上 client 的人不需要网关主人同意就能用
- 网关主人无法知道"现在有谁在用我的 gateway"

v3 的任务：**让"客户端配对 → 配对过的客户端才能用业务接口"成为网关的标准动作，并配套持久化、过期清理、安全防护**。

## 二、目标 / 非目标

### 目标

- **可配对**：客户端首次启动能完成"输入 gateway URL → 配对"全流程；私有模式下"网关主人显式同意"才能配对成功。
- **可鉴权**：配对成功的客户端在业务接口被网关识别为"自己人"；未配对或 key 失效一律 401。
- **可持久**：客户端侧的 gateway URL、唯一键、clientKey 在重启后仍可用。
- **可过期**：网关机主人配置 TTL 后，过期 clientKey 自动清理；不配 TTL 则不清（永久有效）。
- **可观测**：网关机主人能查到"当前配对了多少客户端 / 谁在用"（基于 lastSeenAt 推断在线）。
- **响应规范化**：v3 起所有 endpoint 走统一响应包装 `{data, code, message}`，client 解析逻辑统一。
- **架构预留**：鉴权 / store / 路由设计要为未来"gateway→client 广播"留口子（不实现）。

### 非目标

- **不做 HMAC 签名 / challenge-response**：v3 假设部署在内网或 HTTPS 下，裸 bearer key + HTTPS 即可；v4+ 评估 HMAC。
- **不做 Web UI 解析**：私有模式配对码仅 CLI 解析；不做 Web 端解析（避免引入 Web 鉴权）。
- **不做"管理员后台"**：v3 不引入"查看已配对列表 / 撤销某客户端"的 API；CLI 仅做"解析"动作。
- **不做双向通信 / WebSocket 广播**：v3 架构预留接口，**不实现** v3 阶段。
- **不做 OpenAPI 文档 / Tauri 签名 / Dependabot 开关清理**：v1/v2 遗留，与配对正交，留给后续。
- **不做 v1/v2 兼容**：v2 client 在 v3 gateway 上 `/health` 解析失败（响应格式变了），需升 v3。release notes 注明。
- **不做"网关主人能查看客户端明文 key"**：key 存 hash 不可逆，CLI 不提供"导出 key"功能。

## 三、现状回顾

v2 合入后的状态（commit `ca92c49`）：

- **架构**：`client → gateway → core` 单链路；gateway 仅做路由 + 握手；core 仅做 agent + LLM 调度。
- **持久层**：**完全没有**。gateway 和 core 都是纯内存进程，重启即丢失所有运行时状态。client URL 也未持久化（v2 `App.tsx:60-63` 注释："v3+ 接入 tauri-plugin-store"）。
- **配置**：`gateway/src/config.ts` 5 个字段（PORT/HOST/CORE_URL/LOG_LEVEL/CORS_ORIGINS），全在 env 加载。
- **endpoint**：
  - `GET /health`：`{ok, service, version, schema}` —— liveness + 握手信息
  - `GET /v1/agents`：列 agent 列表（v2 已实现但无鉴权）
  - `POST /v1/chat`：流式聊天（v2 已实现但无鉴权）
- **client**：`App.tsx` 状态机 `PAIRING/HEALTHY/MISMATCH/PAIR_FAILED`；5min heartbeat；`<Settings>` 面板有 URL + 测试按钮，但 URL 不持久化、不在 startup 时强制。
- **依赖**：`fastify / undici / zod / semver` + Tauri 2 + React 18；**无 SQLite / 无 ORM / 无 stronghold**。

这种状态下，"客户端是不是被信任"完全无解。

**架构预留双向通信**：

- `app.websocket` 已在 v2 注册（`@fastify/websocket`）
- v3 不加 WebSocket 业务 endpoint，但所有鉴权逻辑走 `req.clientCtx` 而不是 `req.headers`，将来加 WS endpoint 时可直接复用
- `auth/store.ts` 的 `findByHash` 是 async-friendly（v3 同步、v4+ 可换 async DB 不影响 caller）

### 5.2 统一响应包装（v3 起所有 endpoint）

```ts
// gateway/src/response.ts（新文件）
type ApiResponse<T> =
  | { data: T; code: 0; message: 'ok' }
  | { data: null; code: number; message: string };

export function ok<T>(data: T): ApiResponse<T> {
  return { data, code: 0, message: 'ok' };
}

export function err(code: number, message: string): ApiResponse<null> {
  return { data: null, code, message };
}
```

**所有 endpoint 返回此结构**（含 `/health`、`/pair`、`/pair/status`、`/internal/pair/resolve`、`/v1/*`）。

**`/health` 新形态**：

```json
{
  "data": { "ok": true, "service": "gateway", "version": "0.0.3", "schema": 1 },
  "code": 0,
  "message": "ok"
}
```

liveness 5xx 时：

```json
{
  "data": { "ok": false, "service": "gateway", "version": "0.0.3", "schema": 1 },
  "code": 500,
  "message": "internal_error"
}
```

`/health` 不鉴权，**v2 client 解析 `data.ok` 失败需升级**（v2 才合入几天，强制升级可接受）。

**401 统一响应**：

```json
{ "data": null, "code": 401, "message": "unauthorized" }
```

细分原因放 `message` 字段（`missing_key` / `invalid_key`），不强求 client 区分。

### 5.3 数据模型

```sql
-- 客户端配对记录
CREATE TABLE clients (
  id           TEXT PRIMARY KEY,         -- sha256(clientKey) 的 hex
  key_hash     TEXT NOT NULL UNIQUE,     -- 冗余存储（id 理论可派生，但加唯一约束保护）
  name         TEXT,                     -- 用户在弹出层填的客户端名（可选）
  created_at   INTEGER NOT NULL,         -- 配对时间（epoch ms）
  last_seen_at INTEGER NOT NULL,         -- 最后鉴权时间
  meta         TEXT                      -- JSON: 设备指纹 / IP 列表 / UserAgent（v3 不解析，v4+ 用）
);
CREATE INDEX idx_clients_last_seen ON clients(last_seen_at);

-- 私有模式配对中的 code（解析即删）
CREATE TABLE pairing_codes (
  token       TEXT PRIMARY KEY,          -- 16 字节 base64url 短 token
  client_id   TEXT NOT NULL,             -- 提前绑定 clientKey 的 sha256
  client_name TEXT,                      -- 用户填的 name
  expires_at  INTEGER NOT NULL,          -- 创建时间 + 5 min
  attempts    INTEGER DEFAULT 0          -- 错误尝试次数, 超过 3 次失效
);
CREATE INDEX idx_pairing_codes_expires ON pairing_codes(expires_at);

PRAGMA user_version = 1;
```

**关键决策**：

- 存 `key_hash` 不存明文：DB 泄露也不直接暴露 clientKey
- `clients.id = key_hash`：物理上等价，避免冗余 lookup
- `pairing_codes.client_id` 提前绑定：避免 CLI 解析时再读 client 上传的 name
- `pairing_codes` 解析成功即删：避免堆积（v3 阶段不在 SQL 加 ON DELETE，由应用层处理）
- `meta` 字段 JSON 化：v3 不解析，v4+ 增强用（设备指纹、IP、UA）

**"在线"状态如何保存**：

- 数据模型**无 `online` 字段**
- 网关侧查询时用 `now - last_seen_at < THRESHOLD` 推断（`THRESHOLD` 默认 60s，可配，v3 硬编码）
- 客户端不需要上行 ping（业务请求自然更新 last_seen_at）
- v3 仅暴露 `GET /internal/clients`（127.0.0.1 only）给 CLI 展示"已配对列表 + 在线/离线标记"，**不在 /v1/\* 暴露**

### 5.5 配对 endpoint 决策表

`POST /pair` request: `{ clientKey: string, name?: string, pairKey?: string }`

| 公开模式     | pairKey      | 行为                               | 响应                                                                            |
| ------------ | ------------ | ---------------------------------- | ------------------------------------------------------------------------------- |
| 公开 (true)  | 无 / 错 / 对 | 直接配对（公开模式忽略 pairKey）   | 200 `{ data: { clientKey, name }, code: 0, message: 'ok' }`                     |
| 私有 (false) | 对           | 直接配对（pair key 跨模式 bypass） | 同上                                                                            |
| 私有 (false) | 无 / 错      | 进入 code 流程                     | 202 `{ data: { token, expiresAt, pollUrl }, code: 0, message: 'pair_pending' }` |

**幂等性**（任何模式、任何 pairKey 情况都先跑）：

```
hash = sha256(clientKey)
if exists in clients table:
  return 200 (不再走 pair key 逻辑、不再生成新 token)
```

这覆盖"用户超过轮询时间但网关已配对成功、用户再点配对"场景——重发 POST /pair 时同一个 clientKey 命中 → 直接 200。

**`GET /pair/status?token=...`**：

- token 存在且未过期：200 `{ data: { status: 'PENDING' | 'PAIRED' | 'EXPIRED' }, code: 0, message: 'ok' }`
- PAIRED 时**不返回 clientKey**（client 本就知道自己的）
- token 不存在 / 已过期：404 `{ data: null, code: 404, message: 'token_not_found' }`

**`POST /internal/pair/resolve`**（限 127.0.0.1）：

- request: `{ token: string }`
- 校验：token 存在 + 未过期 + attempts < 3
- 正确：把 `pairing_codes.client_id` 写入 `clients` 表 → 200 `{ data: null, code: 0, message: 'paired' }` → 删除 pairing_codes 行
- 错误：attempts++；超 3 次 → 删除该 token → 404 `{ data: null, code: 404, message: 'token_not_found' }`
- 非 127.0.0.1 访问：403 `{ data: null, code: 403, message: 'forbidden' }`

**`GET /internal/clients`**（限 127.0.0.1）：

- 给 CLI `my-ai-gateway list` 调用
- 200 `{ data: [{ id, name, created_at, last_seen_at, online }], code: 0, message: 'ok' }`
- `online` = `(now - last_seen_at) < 60_000`
- 非 127.0.0.1 访问：403

**幂等性行**（所有决策之上先跑，覆盖"再点配对"场景）：

```
hash = sha256(clientKey)
if exists in clients table:
  return 200 { data: { clientKey, name }, code: 0, message: 'ok' }
  (不查 pairKey、不生成新 token)
```

**CLI 命令**：

```
my-ai-gateway pair --token <token>
```

CLI 进程内部走 HTTP `POST http://127.0.0.1:8787/internal/pair/resolve`（v3 保持现状；v4+ 评估直接读 DB）。

**命名预留**：`my-ai-gateway` bin 字段 + `pair` 子命令。v4+ 改名为 `my-ai` 时只改 `package.json#bin` 字段，不动业务代码。

### 5.6 中间件 + 鉴权流程

```ts
// gateway/src/auth/middleware.ts
import { sha256 } from './hash.js';
import { isPublicPath } from './public-paths.js';

export async function authMiddleware(req, reply) {
  if (isPublicPath(req.url)) return;

  const key = req.headers['x-client-key'];
  if (typeof key !== 'string' || key.length === 0) {
    return reply.code(401).send(err(401, 'missing_key'));
  }

  const hash = sha256(key);
  const client = store.findByHash(hash);
  if (!client) {
    return reply.code(401).send(err(401, 'invalid_key'));
  }

  req.clientCtx = { id: client.id, name: client.name };

  // 异步更新 lastSeenAt，fire-and-forget 不阻塞响应
  setImmediate(() => {
    try {
      store.updateLastSeen(client.id, Date.now());
    } catch (e) {
      app.log.warn({ e, clientId: client.id }, 'updateLastSeen failed');
    }
  });
}
```

**白名单**（`auth/public-paths.ts`）：

```ts
const PUBLIC_PATH_PREFIXES = ['/health', '/pair', '/internal/'];
export function isPublicPath(url: string): boolean {
  return PUBLIC_PATH_PREFIXES.some(p => url === p || url.startsWith(p));
}
```

`/internal/*` 额外检查 `req.ip === '127.0.0.1'`，否则 403。这条检查在 `/internal/pair/resolve` route 自己的 handler 里做（不在中间件里），保持中间件职责单一。

**关键点**：

- 鉴权失败不区分"key 缺"和"key 错"的 HTTP code，统一 401（仅 `message` 字段不同，供调试）
- `setImmediate` 而非 `await`：响应先回，DB 写后台排队
- `updateLastSeen` 失败仅 log warn，不影响当前请求（业务影响小）
- middleware 挂载顺序：在 `cors` 之后、`/v1/*` routes 之前
- `/health` 在白名单中 → 永远不鉴权（monitor / heartbeat 都用）

**Fastify 类型扩展**：

```ts
declare module 'fastify' {
  interface FastifyRequest {
    clientCtx?: { id: string; name: string | null };
  }
}
```

### 5.7 客户端行为

**store schema**（stronghold）：

- `clientKey: string`（首次启动生成 UUID v4 存）
- `gatewayUrl: string`
- `pairKey: string | null`（可选）
- `clientName: string | null`（可选）

**启动流程**：

```
loadConfig() from stronghold
  ├─ 有 clientKey + gatewayUrl → 进入握手
  │   └─ GET /health (X-Client-Key: clientKey)
  │       ├─ 200 → 状态 PAIRED
  │       └─ 5xx / 网络错 → 状态 PAIR_FAILED
  │           └─ 不重试，等用户点"重试"
  └─ 无 clientKey 或无 gatewayUrl → 状态 NEED_PAIR
      └─ 顶部 banner 显示 + 弹出 PairDialog
```

**业务请求 401 → 状态 NEED_REPAIR**（被动感知）：

```
业务请求失败 code=401
  ├─ 清空 stronghold 中的 clientKey（保留 gatewayUrl / pairKey）
  └─ 弹 banner + 弹出 PairDialog
```

**PairDialog 流程**：

```
1. 弹出层表单: [gatewayUrl] [pairKey (可选)] [name (可选)]
2. 用户提交
3. 前置 GET /health (X-Client-Key: clientKey, 如果有)
   ├─ 5xx / 网络错 → 提示"网关不可达", 不进入配对
   └─ 200 → 继续
4. POST /pair { clientKey, name, pairKey }
   ├─ 200 → 存 stronghold, 状态 PAIRED
   ├─ 202 (pair_pending) → 进入轮询
   │   └─ setInterval(poll, 2000), maxAttempts=150 (5min)
   │       ├─ PENDING → 继续
   │       ├─ PAIRED → clearInterval, 状态 PAIRED
   │       └─ 5min 到 / EXPIRED → clearInterval, 提示"配对超时"
   └─ 4xx/5xx → 提示错误, 留弹出层
5. 任何时候用户点"清除配对" → 清空 clientKey + 弹配对
```

**"超过轮询时间再次点击"幂等**：

- 用户在 5min 后点 banner 上的"重新配对"按钮（或弹出层未关 + 用户重提交）
- POST /pair 带同一个 clientKey → 网关按 sha256 命中 → 200 直接配对
- 不重新生成 token，不重新轮询

**轮询停后的"再次点击"**：

- 弹出层未关 + 用户点"重试" → 直接调 POST /pair 走幂等路径
- 弹出层已关 + 用户点 banner 上的"去配对" → 重新打开弹出层（已 pre-fill gatewayUrl/pairKey）→ 提交走幂等

**Banner 上的两个按钮**（`<PairBanner>`）：

- "去配对"：打开 PairDialog（pre-fill stronghold 中的 gatewayUrl/pairKey）
- "清除配对"：清空 stronghold 中的 clientKey（gatewayUrl/pairKey 保留），状态转 NEED_PAIR

**业务请求 401 后的"重新配对"按钮**：

- 同上 "去配对" 按钮，但 PairDialog 上多一个提示条："上次的 clientKey 已失效，请重新配对"

**生成 clientKey**：

- 客户端启动时若 stronghold 中无 → `crypto.randomUUID()` → 存
- UUID v4（128 bit 熵，碰撞概率忽略）
- 生成后**不**可重置（用户主动"清除配对"也不会改 clientKey，因为是同台机器的唯一标识）

### 5.9 过期清理（auth/cleanup.ts）

**启动期**：

```ts
if (cfg.GATEWAY_PAIRING_KEY_TTL && cfg.GATEWAY_PAIRING_KEY_TTL > 0) {
  const intervalSec = Math.max(60, Math.min(3600, Math.ceil(cfg.GATEWAY_PAIRING_KEY_TTL / 10)));
  setInterval(cleanup, intervalSec * 1000);
}
```

**清理逻辑**：

```ts
function cleanup() {
  const ttlMs = cfg.GATEWAY_PAIRING_KEY_TTL * 1000;
  const threshold = Date.now() - ttlMs;
  const deleted = store.deleteExpiredClients(threshold);
  log.info({ deleted }, 'cleanup: removed expired clients');
}
```

**store 索引利用**：`idx_clients_last_seen` 让 `DELETE FROM clients WHERE last_seen_at < ?` 走索引。

**任务间隔策略**：

- 间隔 = `max(60s, min(3600s, TTL/10))`
- 含义：TTL=300s（5min）→ 间隔 60s；TTL=3600s（1h）→ 间隔 360s；TTL=86400s（1d）→ 间隔 3600s
- **不**用 `setTimeout` 链：进程生命周期内固定间隔更简单

**手动触发**：

- CLI 子命令 `my-ai-gateway cleanup`（127.0.0.1 only），方便测试 / 紧急清理

### 5.10 客户端→网关唯一键的三层安全

| 层         | 措施                                | 实施                                                                                                |
| ---------- | ----------------------------------- | --------------------------------------------------------------------------------------------------- |
| 客户端存储 | Tauri stronghold 加密存 OS keychain | `tauri-plugin-stronghold`，key 由 OS 派 生（macOS Keychain / Windows DPAPI / Linux Secret Service） |
| 传输       | HTTPS（部署层）                     | README 显式提示：v3 假设 HTTPS 或内网；明文 HTTP 仅本机场景                                         |
| 网关存储   | SHA-256 hash 不存明文               | `clients.key_hash = sha256(clientKey)`，DB 泄露不暴露 key                                           |

**"防拦截"语义边界**：

- HTTPS 下：TLS 加密传输，header 中的 key 不可被窃听 → 攻击者拿不到
- 局域网明文 HTTP：攻击者拿到 key 后**可以重放**（v3 接受这个风险，README 注明）
- v4+ 评估：加 HMAC 签名（`X-Client-Signature: HMAC(key, ts+path+bodyHash)`）防明文 HTTP 重放

**v3 不做 HMAC 的理由**：

- 部署假设是 HTTPS / 内网，HMAC 是"在明文 HTTP 场景下才需要"的额外层
- 加 HMAC 需要每个业务请求带 ts + signature，对客户端 + 业务逻辑有侵入
- 引入工作量大、收益仅"防明文 HTTP 重放"（已用部署假设替代）

### 5.11 错误处理

| 阶段    | 错误                                                             | 处理                                                          |
| ------- | ---------------------------------------------------------------- | ------------------------------------------------------------- |
| Build   | better-sqlite3 缺 native binding                                 | `pnpm install` 失败                                           |
| Startup | DB 文件不可写                                                    | log error, 进程退出 1                                         |
| Startup | `user_version` 旧                                                | 跑 migration（v3 阶段仅 1，无 migration 代码）                |
| Startup | `GATEWAY_PAIRING_PUBLIC=true` + `GATEWAY_PAIR_KEY` 已配          | warn + 忽略 pairKey                                           |
| Startup | `GATEWAY_PAIRING_KEY_TTL` < 0                                    | 进程退出 1                                                    |
| Runtime | `/pair` 缺 clientKey                                             | 400 `{ code: 400, message: 'missing_client_key' }`            |
| Runtime | `/pair` token 不存在                                             | 404 `{ code: 404, message: 'token_not_found' }`               |
| Runtime | `/pair` token 已过期                                             | 410 `{ code: 410, message: 'token_expired' }`                 |
| Runtime | `/internal/pair/resolve` 错误尝试 > 3                            | 404 `{ code: 404, message: 'token_not_found' }`（删除 token） |
| Runtime | `/internal/*`（含 `pair/resolve` 和 `clients`）非 127.0.0.1 访问 | 403 `{ code: 403, message: 'forbidden' }`                     |
| Runtime | 鉴权失败                                                         | 401 `{ code: 401, message: 'missing_key' \| 'invalid_key' }`  |
| Runtime | DB 写失败（磁盘满）                                              | 5xx 兜底（罕见，仅 updateLastSeen 失败 → log warn）           |
| Runtime | 清理任务删除失败                                                 | log error, 下个 interval 重试                                 |

**核心原则**：build / startup 阶段**loud fail**（早暴露），runtime 阶段**soft fail**（不中断用户）。
