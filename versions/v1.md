## 一、概述

v1 阶段目标：在仓库根目录搭出可运行、可校验、可发布的 monorepo 工程化骨架。具体包含：

- **架构层**：把"客户端 / 网关 / 核心"三个子项目从 0 拼成统一 monorepo。
- **工程化层**：把 lint / format / test / typecheck / hooks / CI 闸门全部补齐。
- **交付层**：通过 GitHub Actions 在 Windows / Linux / macOS 三平台产出可安装包。

本文件是 v1 的"清单 + 决策记录"，不重复 README 的快速开始部分。

```
my-ai/

```

## 四、架构层变更

### 4.1 三个子项目职责切分

| 子项目    | 端口 | 职责                                                       | 关键依赖                               |
| --------- | ---- | ---------------------------------------------------------- | -------------------------------------- |
| `client`  | —    | 桌面 UI + 本地系统集成（OS 自动化后续接 Tauri command）    | Tauri 2 / React 18 / Vite / `undici`   |
| `gateway` | 8787 | 多客户端访问 core 的网关层：路由、鉴权、协议转换、流式代理 | Fastify 4 / undici / pino / zod        |
| `core`    | 8788 | agent 核心：会话、工具、LLM 调用、OS 自动化                | Fastify 4 / pino / zod / AgentRegistry |

数据流：`client → gateway → core → LLM / 工具 / OS 自动化`。

### 4.2 关键设计决策

- **客户端 → 网关**：HTTP + WebSocket（后续可换 SSE / gRPC-Web）。
- **网关 → core**：内部 HTTP（v1 阶段，gRPC 在 v2+ 评估）。
- **LLM 抽象**：通过 `LLMClient` 接口屏蔽 provider；v1 只实现 `MockLLMClient`，`openai-compatible` 留 v2。
- **Agent 模型**：注册表 + 描述符模式，新 agent 只需 `registry.register(new XAgent(llm))`。
- **配置校验**：所有 env 都用 zod schema 校验，启动失败 > 运行期失败。

## 五、工程化层变更

### 5.1 包管理与 Node 版本

- `packageManager: pnpm@10.0.0`：CI 上 `pnpm/action-setup@v6` 自动识别，避免多版本冲突。
- `engines.node: ">=20.0.0"`：本地开发最低门槛 20，CI 锁 24 LTS。
- 工作区：`pnpm-workspace.yaml` 仅声明三个子项目，避免误把脚本目录纳入。

### 5.2 TypeScript 严格模式

`tsconfig.base.json` 关键开关：

| 选项                             | 值     | 目的                                     |
| -------------------------------- | ------ | ---------------------------------------- |
| `strict`                         | `true` | 入口开关                                 |
| `noImplicitOverride`             | `true` | 强制 override 标注                       |
| `noFallthroughCasesInSwitch`     | `true` | 防止 switch 漏写 break                   |
| `noUncheckedIndexedAccess`       | `true` | 数组/对象下标访问推断为 `T \| undefined` |
| `exactOptionalPropertyTypes`     | `true` | 区分 `foo?: T` 与 `foo: T \| undefined`  |
| `isolatedModules`                | `true` | 兼容 SWC / esbuild 单文件转译            |
| `declaration` / `declarationMap` | `true` | 库导出类型 + 源码映射                    |
| `skipLibCheck`                   | `true` | 不查第三方 .d.ts                         |

各子项目 tsconfig 仅覆盖差异：客户端加 DOM/JSX + 路径别名；服务端固定 rootDir/outDir。

### 5.3 ESLint 9 flat config

- 根目录 `eslint.config.mjs` 集中管理，子项目不写本地 config。
- 规则分层：
  - **全局**：`@typescript-eslint/recommended` + `consistent-type-imports` + 显式 `any` 警告。
  - **Node 子项目**（gateway / core）：补 Node globals，放宽 `process.exit`。
  - **客户端**：补 React 18 / Hooks 推荐规则，浏览器 globals。
  - **测试文件**：放宽 `no-explicit-any`、放行 `console`。
- 子项目 `pnpm -r run lint` 通过 pnpm 工作区路径执行，eslint 自动沿父目录链找到根 config。

### 5.4 Prettier + EditorConfig

- `.editorconfig`：缩进 2 空格 / LF / UTF-8 / 末行空行 / 去除行尾空格。
- `.prettierrc.json`：`printWidth: 100`、`singleQuote: true`、`trailingComma: "all"`、`semi: true`。
- `.prettierignore`：构建产物、依赖、Tauri 现场生成物、locks、临时数据。
- 根脚本：`pnpm format`（写入）/ `pnpm format:check`（校验，CI 用）。

### 5.5 Vitest 测试

- 每个子项目一个 `vitest.config.ts`。
- core 6 个用例覆盖 `MockLLMClient`、`AgentRegistry`、`EchoAgent`（隐式通过 registry）、`/health` 路由（fastify.inject）。
- gateway 2 个用例覆盖 `CoreClient` 构造（URL 末尾斜杠处理、默认超时）。
- client 当前 0 个用例，vitest config 已配 `environment: 'jsdom'`、依赖装好，v2 起补组件测试。
- 子项目 `test` 脚本带 `--passWithNoTests`，避免"无测试 = 失败"。

### 5.6 Husky + lint-staged

- Husky 9 通过根 `package.json` 的 `prepare` 脚本在 `pnpm install` 时自动注册 git 钩子。
- `.husky/pre-commit`：跑 `pnpm exec lint-staged`。
- lint-staged 规则：
  - `*.{ts,tsx,js,jsx,mjs,cjs}` → `eslint --fix` + `prettier --write`
  - `*.{json,yml,yaml,md,css,html}` → `prettier --write`
- 端到端冒烟：故意写未用变量 → lint-staged 报错 → 提交被阻断。

### 5.7 CI 闸门

`.github/workflows/build.yml`：

```
quality (ubuntu-latest)
  ├─ typecheck   → pnpm -r run typecheck
  ├─ lint        → pnpm -r run lint
  └─ test        → pnpm -r run test
        ↓
build-macos / build-linux / build-windows
   （needs: quality；先质量后构建，节省 CI 分钟）
```

- `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: 'true'` 显式强制 actions 跑在 Node 24。
- 三平台产物各 2 个：Windows MSI+NSIS、Linux deb+AppImage、macOS app+dmg。

## 六、已知问题

1. **CI 仍有 deprecation 警告**（punycode / url.parse）：
   - 来源：GitHub 官方 `actions/upload-artifact@v5` 内部仍用 Node 18-20 的废弃模块。
   - 影响：仅是 warning，不影响产物。
   - 解决路径：等待 2026-06-16 GitHub 默认切到 Node 24 后自动消失；我方无能为力。

2. **Tauri 产物未签名**：
   - Windows SmartScreen 首次运行需手动放行；macOS 需右键打开。
   - 解决路径：v2+ 引入代码签名证书（需用户提供开发者证书）。

3. **OpenAI 兼容 LLM provider 未实现**：
   - `core/src/llm/index.ts` 在 `provider === 'openai-compatible'` 时抛错。
   - 解决路径：v2 第一个迭代目标。

4. **客户端组件测试空白**：
   - 框架装好（jsdom + Testing Library），无用例。
   - 解决路径：随 chat UI 一起补。

5. **Dependabot 行为与 `.github/dependabot.yml` 不同步**（v1 收尾后追加）：
   - 现象 A：v1 合入 main 后，main 首次出现 `.github/dependabot.yml`（`enabled: false`），Dependabot 立即扫出 8+ 个 outdated 依赖并连开 PR，6 分钟内 CI 跑了 15 次。
   - 现象 B：手动通过 `PUT /repos/{owner}/{repo}/automated-security-fixes` 仅启用 security-updates 后，2 分钟内又开 3 个 version-update PR（vite/vitest/fastify major 升级），与"只开 security"明显不符。
   - 推断根因：GitHub 端 `automated-security-fixes` 端点或仓库层 Dependabot 总开关一旦被 API 激活，就**无视** `dependabot.yml` 里的 `enabled: false`；YAML 字段只在仓库零配置时生效。
   - 临时处理：security-updates API 已 DELETE；`.github/dependabot.yml` 保持 `enabled: false` + 5 个 updates 块（显式声明策略，防御性兜底）。
   - 待处理：v2 启动时搞清楚 Dependabot 三个开关（`dependabot.yml.enabled`、仓库层 `automated-security-fixes`、仓库层 `vulnerability-alerts`）的真正依赖关系；若仍搞不清，最稳妥是**完全删掉 `.github/dependabot.yml`**，让仓库零配置默认零行为。
